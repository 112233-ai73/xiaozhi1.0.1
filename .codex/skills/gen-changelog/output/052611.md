# SD 卡 MP3 软重采样播放改动说明

## 背景

当前日志中曾出现 SD 卡歌曲文件为 `44100 Hz, 2 channel, 16 bit`，而扬声器输出链路仍按 `16000 Hz, 1 channel, 16 bit` 打开。上一份说明中最后关注的状态流为语音识别进入 `LISTENING`，播放完成后由 `SPEAKING -> IDLE` 回到空闲态。

本次工作区相对 `HEAD` 只有 `main/audio/audio_mp3_decode.c` 发生修改，核心目标是让 `/sdcard` 下的 MP3 即使不是 16 kHz 单声道，也能在不改动 `audio_init.c` 输出配置的情况下写入现有播放链路。

## 当前方案：SD 卡文件软重采样到 16 kHz 单声道

当前代码没有切换 ES8311 到 `44100 Hz, 2 channel, 16 bit`，而是在 MP3 解码后增加一层软件处理：

```text
/sdcard/*.mp3
  -> MP3 decoder
  -> 读取原始 MP3 信息
  -> 非 16k/单声道时启用软重采样
  -> 多声道平均为单声道
  -> 按 16 kHz 写入 audio_write()
```

这与“播放 SD 卡内容时切换扬声器到 44.1 kHz 双声道”的目标不同。当前方案是兼容现有 `audio_write()` 的输入格式，避免直接把 44.1 kHz 双声道 PCM 写入 16 kHz 单声道输出设备导致变速、杂音或播放异常。

## 代码侧变化

| 文件 | 变化 | 影响 |
|---|---|---|
| `main/audio/audio_mp3_decode.c` | `mp3_decode_file(FILE *file)` 增加 `enable_resample` 参数 | 播放任务可以按文件路径决定是否启用软重采样 |
| `main/audio/audio_mp3_decode.c` | 新增 `resample_buf`、`resample_phase`、`need_resample`、`in_rate`、`in_channels` | 保存跨 buffer 的重采样状态，降低分块解码时的相位跳变 |
| `main/audio/audio_mp3_decode.c` | 解析 MP3 info 后判断是否需要从原采样率/声道数转换到 `16000Hz 1Ch` | 仅对 `/sdcard` 路径启用转换 |
| `main/audio/audio_mp3_decode.c` | 解码输出写入前执行简单抽样和多声道平均 | 将双声道或更高采样率 PCM 变为单声道 16 kHz PCM |
| `main/audio/audio_mp3_decode.c` | `mp3_player_task()` 使用 `strncmp(file_path, "/sdcard", 7)` 判定 SD 卡路径 | SPIFFS 提示音仍走原播放路径 |
| `main/audio/audio_mp3_decode.c` | 注释掉 `MP3_after_awake` 相关逻辑 | 播放请求创建后不再在该位置主动设置唤醒状态 |

## 详细设计

### 路径判定

播放任务打开文件成功后，通过路径前缀判断是否为 SD 卡文件：

```c
bool enable_resample = false;
if (strncmp(file_path, "/sdcard", 7) == 0) {
    enable_resample = true;
}
```

因此当前影响范围只覆盖 `/sdcard` 开头的文件。`/spiffs` 下的提示音、启动音和命令反馈音不启用软重采样。

### MP3 信息读取

第一次成功解码后，代码通过 `esp_audio_dec_get_info()` 获取：

```text
sample_rate
channel
bits_per_sample
```

当 `enable_resample == true` 且输入不是 `16000 Hz / 1 channel` 时，设置 `need_resample = true` 并申请 `resample_buf`。

### 软重采样与降混音

当前重采样逻辑为简单抽样：

```text
step = in_rate / 16000.0
resample_phase += step
```

声道转换使用多声道求平均：

```text
mono = (ch0 + ch1 + ... + chN) / channel_count
```

处理完成后写入：

```c
audio_write((uint8_t *)resample_buf, out_frames * sizeof(int16_t));
```

### 状态流

当前播放状态流保持为：

```text
audio_mp3_play_file_async()
  -> 创建 mp3_player_task
  -> 可选暂停 MultiNet
  -> 打开 MP3 文件
  -> /sdcard 路径启用软重采样
  -> mp3_decode_file()
  -> 播放完成 com_status_change(IDLE)
  -> 恢复 MultiNet
```

如果播放期间被停止：

```text
s_mp3_stop_requested = true
  -> 解码循环退出
  -> MP3 playback stopped
  -> com_status_change(IDLE)
```

## 与原目标的差异

| 原目标 | 当前代码实现 | 说明 |
|---|---|---|
| SD 卡播放时 ES8311 切换到 `44100 Hz, 2 channel, 16 bit` | SD 卡播放时软件转换到 `16000 Hz, 1 channel, 16 bit` | 当前没有调用 `esp_codec_dev_close/open` 重新配置播放 codec |
| 其他时候保持 `16000 Hz, 1 channel, 16 bit` | 其他路径保持原 `audio_write()` 直写 | SPIFFS 提示音不受影响 |
| 按歌曲原格式输出 | 按现有扬声器输出格式喂数据 | 更稳妥，但不是高保真播放路径 |

## 可删除或需要清理的代码

| 项目 | 所在文件 | 原因 |
|---|---|---|
| `// ... 下方的 ... 保持不变 ...` 注释 | `main/audio/audio_mp3_decode.c` | 这是说明性占位注释，不适合作为正式工程代码保留 |
| 被注释掉的 `MP3_after_awake` 逻辑 | `main/audio/audio_mp3_decode.c` | 如果确认不再需要唤醒后播放状态联动，应删除；如果仍需要，应恢复为明确实现 |
| 文件末尾缺失换行 | `main/audio/audio_mp3_decode.c` | 建议补齐，避免后续 diff 噪声 |
| 当前软重采样中的中文注释乱码 | `main/audio/audio_mp3_decode.c` | 说明文件编码或编辑器编码不一致，建议统一为 UTF-8 |

## 风险评估

| 风险点 | 触发条件 | 影响 | 建议验证 |
|---|---|---|---|
| 简单抽样质量有限 | 44.1 kHz 或其他采样率降到 16 kHz | 高频混叠、音质下降 | 用 44.1 kHz 双声道歌曲实听，确认是否可接受 |
| 未检查 `resample_buf` 分配失败 | `malloc(out_buf_size)` 返回 `NULL` | 会绕过重采样并可能没有明确错误日志 | 增加内存不足日志或失败返回 |
| 只根据 `/sdcard` 前缀启用 | 路径为 `/sdcardxxx` 也会匹配 | 可能误判路径 | 建议改为匹配 `/sdcard/` |
| 没有处理 bits 非 16 的转换 | MP3 解码输出不是 16 bit | 当前只记录或旁路，不做位深转换 | 用不同 MP3 文件验证 `bits_per_sample` |
| `MP3_after_awake` 逻辑被注释 | 唤醒后播放依赖该变量 | 可能影响唤醒状态恢复 | 回归唤醒后播报流程 |
| `stop_play_mp3()` 忙等 | 播放任务长时间未退出 | 占用 CPU | 建议后续改为延时等待或事件同步 |

## 迁移建议

如果最终目标仍是“SD 卡播放时切换 ES8311 到 44.1 kHz 双声道”，建议后续改为硬件输出格式切换方案：

1. 在 `audio_init.c` 中封装 `audio_set_playback_format(sample_rate, channel, bits)`。
2. 播放 SD 卡文件前关闭并重新打开 `play_dev_handle` 为 `44100/2/16`。
3. SD 卡播放结束或失败后恢复为 `16000/1/16`。
4. 用互斥锁保护 `audio_write()` 和 codec 重新打开过程。
5. 保留当前软重采样作为可选 fallback，而不是默认路径。
