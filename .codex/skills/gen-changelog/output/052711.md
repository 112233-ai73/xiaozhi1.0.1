# WebSocket 在线语音链路改造说明

## 背景

本次改动围绕“小智唤醒后进入在线交流”的链路展开。当前工程保留了 `audio_offline_sr_handler` 的本地唤醒能力，唤醒后不再播放本地唤醒提示音，而是异步建立 WebSocket 连接，完成 `hello` 握手并拿到 `session_id` 后发送唤醒消息。

改造后的目标是：

1. 唤醒词触发后异步连接 WebSocket，不阻塞语音识别任务。
2. WebSocket 连接成功后设置 `is_wsline=true`，并开始在线语音交流。
3. 唤醒后服务端返回的第一段回复音频不播放。
4. 用户说话阶段上传 Opus 数据，服务端回复阶段停止上传并进入 `SPEAKING`。
5. 服务端返回的二进制 Opus 音频进入本地解码播放链路。

当前工作区相对 `HEAD` 涉及 7 个文件：

| 文件 | 变更规模 | 主要内容 |
|---|---:|---|
| `main/audio/audio_online_sr_handle.c` | `~276 行` | 新增在线 WebSocket 回调、异步连接任务、Opus 上传任务、TTS 状态处理 |
| `main/audio/audio_online_sr_handle.h` | `~4 行` | 暴露在线初始化、异步启动和就绪检查接口 |
| `main/audio/audio_sr.c` | `~21 行` | `LISTENING/SPEAKING` 状态下控制 WebSocket 收音开始、停止和上传门控 |
| `main/audio/audio_offline_sr_handler.c` | `~14 行` | 唤醒后改为启动在线链路，不再播放本地唤醒 MP3 |
| `main/bsp/bsp_ws.c` | `~34 行` | WebSocket 连接状态、回调注册、session 清理和空指针保护 |
| `main/bsp/bsp_ws.h` | `~5 行` | 新增 WebSocket 回调注册和连接状态查询接口 |
| `main/main.c` | `~4 行` | 启动阶段初始化在线音频链路 |

## 推荐方案：唤醒驱动的在线语音状态机

本次实现采用“离线唤醒 + 在线对话”的组合方案：

```text
离线 MultiNet 唤醒
  -> audio_online_start_async()
  -> bsp_ws_start()
  -> bsp_ws_send_hello()
  -> 收到 hello/session_id
  -> bsp_ws_send_wakeup_word()
  -> IDLE
  -> VAD 检测到用户说话
  -> LISTENING + listen/start + Opus 上传
  -> VAD 静音超时
  -> listen/stop + IDLE
  -> 服务端 tts/start
  -> SPEAKING + 停止上传
  -> 服务端二进制 Opus 下行
  -> 本地解码播放
  -> 服务端 tts/stop
  -> IDLE
```

这个方案让唤醒和在线交流解耦：唤醒仍由本地模型完成，WebSocket 连接、握手、音频上传和回复播放由 `audio_online_sr_handle` 统一承接。

## 为什么选择这个方案

| 原问题 | 新方案 | 影响 |
|---|---|---|
| 唤醒后只能播放本地 `/spiffs/105.mp3` | 唤醒后调用 `audio_online_start_async()` | 唤醒行为直接进入在线能力 |
| WebSocket 回调没有统一接入在线音频模块 | `bsp_ws_set_callbacks()` 注册文本和二进制回调 | BSP 层只负责收发，业务解析下沉到 audio 在线模块 |
| 服务端回复时仍可能继续上传麦克风数据 | `tts/start` 切到 `SPEAKING`，并发送 `listen/stop` | 回复期间停止上传，避免把扬声器声音再次送给后端 |
| WebSocket 文本数据没有保证以 `\0` 结尾 | `copy_ws_text()` 复制并补零后再交给 cJSON | 降低 JSON 解析越界风险 |
| 连接状态和会话状态复用旧值 | `bsp_ws_start()` 清理连接/session bit 和 `session_id` | 避免重连时误用旧 session |

## 架构总览

### 模块职责

| 模块 | 职责 |
|---|---|
| `audio_offline_sr_handler.c` | 只保留唤醒触发入口，唤醒后启动在线链路 |
| `audio_online_sr_handle.c` | 在线音频编解码、WebSocket 回调、异步连接、Opus 上传、下行播放 |
| `audio_sr.c` | AFE/VAD/MultiNet 识别，同时在 `LISTENING` 时把 AFE 数据写入上传 ringbuffer |
| `bsp_ws.c` | WebSocket 客户端生命周期、事件标志位、协议消息发送 |
| `com_status.c` | 维护 `IDLE/LISTENING/SPEAKING` 和唤醒超时 |

### 任务与数据流

```text
audio_detect_task
  -> afe_handle->fetch()
  -> VAD_SPEECH 时切 LISTENING
  -> LISTENING 且 is_wsline=true 时写 g_audio_out_ringbuf

audio_encode task
  -> 从 g_audio_out_ringbuf 读取 PCM
  -> 编码为 Opus
  -> 写 encode_output_buffer

online_send_task
  -> 从 encode_output_buffer 读取 Opus
  -> 仅在 is_wsline && com_status == LISTENING 时 bsp_ws_send_opus()

WebSocket binary callback
  -> 非唤醒回复丢弃阶段时写 decode_input_buffer
  -> audio_decode task 解码
  -> online_play_task 调用 audio_write()
```

## 详细设计

### 1. 唤醒后异步启动 WebSocket

`audio_offline_sr_handler.c` 中 `command_id` 为 `0/1/2` 的唤醒分支现在执行：

```text
stop_play_mp3()
com_set_awake(true)
audio_online_start_async()
```

旧的 `com_status_change(SPEAKING)` 和 `audio_mp3_play_file_async("/spiffs/105.mp3")` 被移除，表示唤醒本身不再进入本地播报状态，而是交给在线链路完成后续对话。

### 2. 在线模块初始化

`main.c` 在 `app_sr_start()` 之后调用：

```c
ESP_ERROR_CHECK(audio_online_init());
```

这样可以保证 `app_sr_get_audio_ringbuf()` 已经创建完成，在线编码器能够注册 AFE 输出 ringbuffer。

`audio_online_init()` 负责：

1. 创建 `audio_online_sr_handle_t`。
2. 启动 Opus 编码、解码、播放和上传任务。
3. 初始化 WebSocket。
4. 注册 WebSocket 文本和二进制回调。

### 3. WebSocket 握手与 session 管理

`audio_online_start_async()` 会创建 `online_ws_task`，该任务负责：

```text
如果未连接 -> bsp_ws_start()
连接成功 -> bsp_ws_send_hello()
收到 hello/session_id -> 设置 WEBSOCKET_SESSION_ID_BIT
设置 s_drop_wakeup_reply=true
发送 bsp_ws_send_wakeup_word()
切回 IDLE 等待用户说话
```

`bsp_ws_start()` 新增了连接前状态清理：

```text
清除 WEBSOCKET_CONNECTED_BIT
清除 WEBSOCKET_SESSION_ID_BIT
清空 session_id
```

这样可以避免断线重连时沿用旧 session 状态。

### 4. TTS 回复期间停止上传

`handle_tts_message()` 收到服务端：

```json
{"type":"tts","state":"start"}
```

时，如果不是唤醒回复丢弃阶段，会执行：

```text
如果当前是 LISTENING 且 WebSocket 在线 -> bsp_ws_send_stop_listen()
com_status_change(SPEAKING)
```

`audio_sr.c` 也增加了 `SPEAKING` 门控：

| 位置 | 行为 |
|---|---|
| `switch_to_listening()` | 如果当前是 `SPEAKING`，直接返回，不允许 VAD 抢回 `LISTENING` |
| `handle_vad_state()` | 如果当前是 `SPEAKING`，返回 `false`，不执行上传和 MultiNet 检测 |
| `online_send_task` | 只有 `is_wsline && com_status == LISTENING` 时才发送 Opus |

因此服务端回复过程中不会继续上传麦克风数据。

### 5. 唤醒回复丢弃机制

唤醒后发送 `wakeup_word` 前会设置：

```c
s_drop_wakeup_reply = true;
```

在该标志为 `true` 时：

| 数据类型 | 处理方式 |
|---|---|
| 服务端 `tts/start` | 不进入 `SPEAKING` |
| 服务端二进制 Opus | 直接丢弃，不写入解码队列 |
| 服务端 `tts/stop` | 清除 `s_drop_wakeup_reply` |

这样可以满足“成功拿到 `session_id` 后发送唤醒，但是不会播放后台对唤醒后的回复”的需求。

## 吞吐量、资源与风险评估

| 风险点 | 触发条件 | 影响 | 建议验证 |
|---|---|---|---|
| `session_id` 缓冲区较短 | 服务端返回超过当前数组长度的 session | `strlcpy` 会截断 session，可能导致后续消息 session 不完整 | 确认服务端 session 长度，必要时扩大 `session_id` |
| 唤醒回复丢弃标志依赖 `tts/stop` | 服务端没有发送 `tts/stop` | `s_drop_wakeup_reply` 可能一直为 `true`，后续音频也被丢弃 | 联调确认唤醒回复一定有 `tts/stop`，或增加超时兜底 |
| `bsp_ws_start()` 阻塞等待连接 | 网络不可用或 WebSocket 事件未返回 | `online_ws_task` 阻塞，但不会阻塞主识别任务 | 断网场景下验证任务和状态是否可恢复 |
| 回复播放与本地 VAD 同时运行 | 扬声器声音被麦克风采集 | 当前 `SPEAKING` 门控会阻止上传，但 AFE 仍在 fetch | 上板确认回复期间无 `listen/start` 和 Opus 上传 |
| 上传 ringbuffer 堆积 | `SPEAKING` 或离线期间编码器持续产出但发送任务不发送 | 可能出现旧帧在重新 `LISTENING` 后被发送 | 需要实测，必要时在状态切换时清空 ringbuffer |

## 与当前方案的映射

| 原行为 | 新行为 | 说明 |
|---|---|---|
| 唤醒后播放本地提示音 | 唤醒后异步连接 WebSocket 并发送唤醒消息 | 唤醒从本地反馈切换为在线入口 |
| WebSocket 初始化入口在 `main.c` 中注释 | `audio_online_init()` 内部初始化 WebSocket | 在线模块统一管理 WebSocket 回调 |
| WebSocket data 回调直接调用函数指针 | 回调前检查 `NULL` | 避免未注册回调时崩溃 |
| `LISTENING` 只用于本地识别 | `LISTENING` 同时表示在线上传窗口 | 只有该状态才上传 Opus |
| 服务端回复不明确控制本地状态 | `tts/start -> SPEAKING`，`tts/stop -> IDLE` | 让在线回复进入统一状态机 |
| 回复过程中 VAD 可能再次进入监听 | `SPEAKING` 中禁止切回 `LISTENING` | 防止后台回复期间继续上传 |

## 迁移步骤

1. 保留当前离线唤醒词配置，确认 `command_id=0/1/2` 能够进入唤醒分支。
2. 确认 `app_main()` 中 `app_sr_start()` 成功后调用 `audio_online_init()`。
3. 配置真实 WebSocket 地址、token 和服务端协议参数。
4. 唤醒小智，观察是否出现 WebSocket 连接成功、`hello`、`session_id` 日志。
5. 确认唤醒后服务端第一段回复不会播放。
6. 唤醒后说话，确认进入 `LISTENING`、发送 `listen/start`、上传 Opus。
7. 停止说话，确认 VAD 超时后发送 `listen/stop`。
8. 服务端回复时确认进入 `SPEAKING`，且期间没有继续上传 Opus。
9. 服务端发送 `tts/stop` 后确认状态回到 `IDLE`，下一轮对话可继续。

## 后续接入其他智能体的改造方式

后续如果需要接入其他基于小智二次开发的智能体，建议不要直接在 `audio_sr.c`、`audio_offline_sr_handler.c` 中散落不同平台的判断，而是在 WebSocket 协议层上方增加“智能体适配层”。核心目标是把“连接参数、握手字段、唤醒消息、监听消息、TTS/STT/LLM/IOT 消息解析”从当前小智协议中抽出来，形成可切换的 agent profile。

### 建议新增的抽象

| 抽象 | 建议位置 | 职责 |
|---|---|---|
| `agent_profile_t` | `main/app` 或 `main/audio` | 描述当前智能体的名称、WebSocket URL、token、设备信息、协议版本 |
| `agent_protocol_ops_t` | 新增 `app_agent_protocol.c/.h` | 封装 `send_hello`、`send_wakeup`、`send_listen_start`、`send_listen_stop`、`parse_text` |
| `agent_runtime_t` | 新增 `app_agent_runtime.c/.h` | 保存当前 `session_id`、是否丢弃唤醒回复、智能体在线状态 |
| 配置来源 | NVS、menuconfig、串口协议或 SD 卡配置文件 | 决定启动时选择哪个智能体 |

建议结构：

```c
typedef struct {
    const char *name;
    const char *ws_url;
    const char *token;
    const char *protocol_version;
    const char *device_id;
} agent_profile_t;

typedef struct {
    esp_err_t (*send_hello)(const agent_profile_t *profile);
    esp_err_t (*send_wakeup)(const agent_profile_t *profile, const char *session_id);
    esp_err_t (*send_listen_start)(const char *session_id);
    esp_err_t (*send_listen_stop)(const char *session_id);
    esp_err_t (*parse_text)(const char *text, int len);
} agent_protocol_ops_t;
```

### 需要修改的现有位置

| 现有位置 | 当前职责 | 接入多智能体后的修改方式 |
|---|---|---|
| `bsp_ws.c` | 固定 WebSocket 地址、token、发送小智协议 JSON | 保留底层收发能力，URL/token 改为由 `agent_profile_t` 注入 |
| `audio_online_sr_handle.c` | 解析 `hello/tts/stt/llm/iot`，发送唤醒和收听控制 | 文本解析和消息发送改为调用 `agent_protocol_ops_t` |
| `audio_online_start_async()` | 默认连接当前小智服务 | 增加 `audio_online_start_agent_async(agent_id)` 或传入 `agent_profile_t` |
| `session_id` 全局变量 | 保存当前小智会话 | 移到 `agent_runtime_t`，避免不同智能体 session 规则不一致 |
| `s_drop_wakeup_reply` | 控制唤醒后首段回复丢弃 | 移到 runtime，并允许不同智能体配置是否丢弃唤醒回复 |
| `app_usart_protocol.c` | 当前主要处理串口命令 | 可增加“切换智能体”命令，写入 NVS 后重连 WebSocket |

### 协议适配点

不同二开智能体通常会保留“小智式”的音频二进制上传和 TTS 下行，但 JSON 字段可能不同。接入前需要逐项确认：

| 协议点 | 当前实现 | 接入其他智能体时需要确认 |
|---|---|---|
| WebSocket URL | 写在 `bsp_ws.c` 的配置中 | 是否每个智能体 URL 不同，是否需要 path/query 参数 |
| 认证方式 | 当前使用 token | token 放 header、query 还是 JSON hello 字段 |
| hello 消息 | `bsp_ws_send_hello()` | 字段名、设备 ID、能力声明、音频格式声明是否一致 |
| session 返回 | `type=hello` + `session_id` | session 字段名、长度、是否每轮对话变化 |
| 唤醒消息 | `bsp_ws_send_wakeup_word()` | 是否需要携带唤醒词、agent id、语言、地理位置等 |
| 监听开始 | `bsp_ws_send_start_listen()` | 是否叫 `listen/start`，是否需要 VAD 模式或音频参数 |
| 监听停止 | `bsp_ws_send_stop_listen()` | 是否叫 `listen/stop`，是否需要结束原因 |
| 音频上传 | `bsp_ws_send_opus()` | Opus 采样率、帧长、声道数、是否需要二进制帧头 |
| TTS 状态 | `type=tts` + `state=start/stop` | 是否使用其他事件名，例如 `speech/start`、`response/audio/done` |
| 文本结果 | `stt/llm/iot` | 是否需要把 LLM 文本展示、串口上报或触发本地控制 |

### 推荐迁移步骤

1. 先保持当前小智协议不变，把现有 `bsp_ws_send_hello()`、`bsp_ws_send_wakeup_word()`、`bsp_ws_send_start_listen()`、`bsp_ws_send_stop_listen()` 包装成一个默认 `xiaozhi_protocol_ops`。
2. 新增 `agent_profile_t`，先只放当前小智服务的 URL、token、设备 ID，确认行为不变。
3. 把 `audio_online_ws_text_cb()` 中的 JSON 解析移动到 `agent_protocol_ops_t.parse_text()`，由适配层返回统一事件，例如 `AGENT_EVT_HELLO`、`AGENT_EVT_TTS_START`、`AGENT_EVT_TTS_STOP`。
4. `audio_online_sr_handle.c` 只处理统一事件，不直接关心不同智能体的原始 JSON 字段。
5. 为第二个智能体新增一份 `agent_protocol_ops_t`，只改协议构造和解析，不改音频上传、VAD、播放链路。
6. 增加配置入口：优先建议用 NVS 保存 `agent_id`，串口命令或菜单配置负责切换。
7. 切换智能体时执行 `bsp_ws_stop()`，清空 runtime session，再按新 profile 重新 `audio_online_start_async()`。

### 推荐状态流

多智能体接入后，底层状态机仍保持当前设计：

```text
IDLE
  -> 唤醒
  -> 选择当前 agent_profile
  -> WebSocket connect
  -> agent send_hello
  -> agent session ready
  -> agent send_wakeup
  -> IDLE
  -> VAD speech
  -> LISTENING + agent listen_start + Opus upload
  -> VAD idle
  -> agent listen_stop + IDLE
  -> agent tts_start
  -> SPEAKING + stop upload
  -> agent tts_stop
  -> IDLE
```

这样可以保证不同智能体只替换“协议适配层”，不破坏已经打通的本地音频链路。

### 验证清单

| 场景 | 预期 |
|---|---|
| 默认小智 profile | 行为与当前实现一致 |
| 切换到新智能体 profile | WebSocket 连接到新 URL，hello 字段符合新协议 |
| 新智能体返回 session | runtime 正确保存 session，不截断 |
| 唤醒后首段回复 | 根据 profile 配置决定播放或丢弃 |
| 用户说话 | `LISTENING` 期间仍只上传 Opus |
| 智能体回复 | `tts_start` 映射到 `SPEAKING`，并停止上传 |
| 切回小智 profile | 旧连接关闭，新 session 重新建立 |

## 后续建议

1. 给 `s_drop_wakeup_reply` 增加超时兜底，避免服务端异常漏发 `tts/stop` 后一直丢弃音频。
2. 确认 `session_id` 的最大长度，必要时把全局缓冲区从 9 字节扩大。
3. 在 `LISTENING -> SPEAKING` 或 `SPEAKING -> IDLE` 时评估是否需要清空编码输出 ringbuffer，避免旧 Opus 帧残留。
4. 将 WebSocket token 从测试值替换为正式配置来源。
