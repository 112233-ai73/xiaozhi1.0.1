# SD 卡 MP3 播放与离线语音歌曲控制改动说明

## 背景

本次改动围绕 ESP32-S3 端的本地 MP3 播放链路展开，目标是让离线语音命令除了播放 SPIFFS 分区中的固定提示音外，还能扫描并播放 SD 卡中的歌曲文件。为支持这条链路，代码同步调整了 MP3 播放接口的路径语义、SD 卡目录扫描、离线语音命令映射、启动初始化顺序和命令词表。

当前工作区相对 `HEAD` 涉及 12 个文件：

| 文件 | 变更规模 | 主要内容 |
|---|---:|---|
| `main/audio/audio_mp3_decode.c` | `~41 行` | MP3 播放接口从文件名改为绝对路径，按路径判断是否挂载 SPIFFS |
| `main/bsp/bsp_sdcard.c` | `~51 行` | 新增 SD 卡 MP3 文件扫描和完整路径缓存 |
| `main/bsp/bsp_sdcard.h` | `~9 行` | 导出 SD 卡歌曲列表、数量和扫描接口 |
| `main/audio/audio_offline_sr_handler.c` | 新增/重命名 | 修正文件名拼写，并新增 SD 卡歌曲播放、上一首、下一首命令 |
| `main/audio/audio_offlline_sr_handler.c` | 删除 | 删除拼写错误的旧文件名 |
| `main/audio/audio_offline_sr_handler.h` | `~1 行` | 引入 SD 卡模块头文件 |
| `main/com/command_word.c` | `~5 行` | 新增播放歌曲、上一首、下一首命令词，并修复字符串拼接问题 |
| `main/main.c` | `~5 行` | 启动时初始化 SD 卡并扫描 `/sdcard` |
| `main/app/app_usart_protocol.c` | `~26 行` | 串口触发 MP3 播放时改用 `/spiffs/...` 绝对路径 |
| `main/com/com_status.c/.h` | `~4 行` | 睡眠提示音改为绝对路径，新增 `is_wsline` 开关 |
| `main/audio/audio_sr.c` | `~2 行` | 音频 ringbuffer 输出受 `is_wsline` 控制 |

## 推荐方案：统一绝对路径的本地音频播放

新的播放链路把 MP3 播放入口统一为绝对路径：

```text
SPIFFS 提示音: /spiffs/105.mp3
SD 卡歌曲:   /sdcard/song01.mp3
```

`audio_mp3_play_file_async()` 不再自己拼接 `/spiffs/<file_name>`，而是直接使用调用方传入的完整路径。这样 SPIFFS 和 SD 卡可以共用同一个播放函数，调用方负责决定音频文件来自哪个挂载点。

## 为什么选择这个方案

| 原问题 | 新方案 | 影响 |
|---|---|---|
| MP3 播放函数默认把文件名拼到 `/spiffs` 下 | 调用方传入绝对路径 | 支持 `/spiffs` 和 `/sdcard` 两类来源 |
| SD 卡歌曲无法被语音命令引用 | 启动时扫描 SD 卡并缓存完整路径 | `bo fang ge qu` 可直接播放歌曲列表当前项 |
| 离线 handler 文件名 `audio_offlline` 拼写错误 | 改为 `audio_offline` | CMake 引用更一致，避免后续维护误判 |
| 新增命令词缺少逗号时会被 C 字符串自动拼接 | 命令词末尾补齐逗号并新增 3 条独立命令 | 避免 MultiNet 加载时报 `invalid command` |

```text
app_main()
```

### SD 卡歌曲播放路径

```text
SD 卡挂载成功
  -> sd_list_files("/sdcard")
       -> readdir()
       -> 过滤 .mp3/.MP3
       -> 保存 /sdcard/<name>.mp3 到 sd_file_names[]

离线语音识别命令
  -> command_id 80/81/82
  -> audio_mp3_play_file_async(sd_file_names[music_num])
  -> fopen("/sdcard/<name>.mp3", "rb")
  -> MP3 解码并 audio_write()
```

```text
audio_detect_task
  -> MultiNet 检测命令
  -> xQueueSend(g_result_que)

sr_handler_task
  -> 消费 command_id
  -> 固定提示音走 /spiffs/*.mp3
  -> 歌曲播放走 sd_file_names[music_num]

mp3_player_task
  -> 打开绝对路径
  -> 解码 MP3
  -> 播放完成后 com_status_change(IDLE)
```

## 详细设计

### MP3 播放路径语义调整

`main/audio/audio_mp3_decode.c` 将 `mp3_play_request_t` 中的 `file_name` 改为 `file_path`，最大长度从 `64` 扩到 `128`。播放入口现在要求传入以 `/` 开头的绝对路径：

```c
audio_mp3_play_file_async("/spiffs/105.mp3");
audio_mp3_play_file_async("/sdcard/song01.mp3");
```

当路径前缀是 `/spiffs/` 时，播放模块会调用 `mount_storage_partition()` 挂载 SPIFFS；当路径是 `/sdcard/...` 时，不再尝试挂载 SPIFFS，直接交给 VFS 打开文件。

这意味着 SD 卡挂载责任前移到启动流程或 SD 卡模块，播放模块只负责消费已可访问的 VFS 路径。

### SD 卡 MP3 扫描

`main/bsp/bsp_sdcard.c` 新增全局歌曲列表：

```c
char sd_file_names[MAX_FILES_TO_LIST][MAX_FILENAME_LEN];
int sd_file_count;
```

`sd_list_files()` 负责打开指定目录，过滤 `.mp3` 后缀文件，并保存完整路径：

```text
/sdcard/001.mp3
/sdcard/song01.mp3
```

当前限制：

| 限制项 | 当前值 | 说明 |
|---|---:|---|
| 最大文件数 | `30` | 超过后停止记录 |
| 单个路径缓存 | `300` 字节 | SD 卡列表可保存较长路径 |
| 播放请求路径 | `128` 字节 | 真正播放时仍受 `audio_mp3_decode.c` 限制 |

### 离线语音命令扩展

`main/com/command_word.c` 新增三条命令：

| command_id | 命令词 | 行为 |
|---:|---|---|
| `80` | `bo fang ge qu` | 播放当前歌曲 |
| `81` | `xia yi shou` | 切到下一首并播放 |
| `82` | `shang yi shou` | 切到上一首并播放 |

`main/audio/audio_offline_sr_handler.c` 新增 `music_num` 保存当前歌曲索引，并在 `case 80/81/82` 中调用：

```c
audio_mp3_play_file_async(sd_file_names[music_num]);
```

上一首/下一首使用边界保护，避免索引越过 `0` 或 `sd_file_count - 1`。

### SPIFFS 提示音调用统一

多个模块中的提示音调用从相对文件名改为绝对路径，例如：

```text
105.mp3          -> /spiffs/105.mp3
welcome.mp3      -> /spiffs/welcome.mp3
107.mp3          -> /spiffs/107.mp3
```

涉及位置包括：

| 模块 | 调整内容 |
|---|---|
| `main/main.c` | 启动提示音改为 `/spiffs/107.mp3` |
| `main/com/com_status.c` | 睡眠提示音改为 `/spiffs/106.mp3` |
| `main/app/app_usart_protocol.c` | 串口命令播放 `200.mp3` 到 `208.mp3` 时使用绝对路径 |
| `main/audio/audio_offline_sr_handler.c` | 离线语音反馈音全部使用 `/spiffs/...` |

### 音频 ringbuffer 输出开关

`main/com/com_status.c/.h` 新增 `is_wsline` 全局开关。`main/audio/audio_sr.c` 在向 `g_audio_out_ringbuf` 写入 AFE 数据前增加判断：

```c
if (g_audio_out_ringbuf != NULL && is_wsline == true)
```

这表示 WebSocket/在线音频输出链路可以通过 `is_wsline` 控制是否接收离线识别期间的音频帧，避免默认持续写入 ringbuffer。

## 风险评估

| 风险点 | 触发条件 | 影响 | 建议验证 |
|---|---|---|---|
| `sd_file_count == 0` 时播放歌曲 | SD 卡无 MP3 或扫描失败后说“播放歌曲” | `sd_file_names[0]` 为空路径，播放返回参数错误或打开失败 | 在 `case 80/81/82` 前增加空列表保护 |
| SD 卡文件名编码异常 | SD 卡包含中文、特殊符号或 FatFs 编码不匹配 | `readdir()` 返回短文件名或乱码路径，`fopen()` 失败 | 先使用 `001.mp3`、`002.mp3` 等 ASCII 短文件名验证 |
| MP3 格式与播放设备不匹配 | SD 卡歌曲为 44.1kHz/双声道，而播放设备为 16kHz/单声道 | 播放速度、音调或音质异常 | 统一转码为 `16kHz/16bit/mono`，或后续加入重采样 |
| 播放请求路径长度不足 | SD 卡路径超过 `MP3_FILE_PATH_MAX_LEN=128` | 播放入口返回 `ESP_ERR_INVALID_SIZE` | 缩短文件名，或统一扩大播放请求路径缓存 |
| `stop_play_mp3()` 忙等 | 播放任务异常无法及时退出 | 调用方可能阻塞 CPU | 后续改为事件/信号量等待，并加超时 |
| 启动时 SD 卡挂载失败 | 未插卡或 SPI 接线异常 | `ESP_ERROR_CHECK(sd_list_files())` 可能导致启动中止 | 产品模式下建议降级为告警，不阻断语音识别 |

## 与当前方案的映射

| 旧实现 | 新实现 | 说明 |
|---|---|---|
| `audio_mp3_play_file_async("105.mp3")` | `audio_mp3_play_file_async("/spiffs/105.mp3")` | 播放接口从文件名语义切换为绝对路径语义 |
| 播放模块固定拼接 `/spiffs` | 调用方传入 `/spiffs` 或 `/sdcard` | 播放来源由调用方决定 |
| 离线命令只控制设备和提示音 | 离线命令新增歌曲播放控制 | `command_id 80/81/82` 对应当前/下一首/上一首 |
| SD 卡初始化后只做测试读写 | SD 卡初始化后扫描 MP3 列表 | SD 卡进入业务播放链路 |
| 新增命令词可能被字符串拼接 | 每条命令词独立逗号分隔 | 避免 MultiNet 命令格式错误 |

## 迁移步骤

1. 保持所有 SPIFFS 提示音调用使用 `/spiffs/<file>.mp3` 绝对路径。
2. SD 卡歌曲建议先统一命名为 `001.mp3`、`002.mp3`、`003.mp3`，避免中文和特殊字符。
3. SD 卡歌曲建议转码为 `16kHz/16bit/mono` MP3，以匹配当前播放设备配置。
4. 开机后确认日志出现 SD 卡挂载成功和 `目录读取完成`。
5. 依次测试 `bo fang ge qu`、`xia yi shou`、`shang yi shou`，观察是否打开 `/sdcard/...` 路径。
6. 若后续要支持普通 44.1kHz 双声道歌曲，需要在播放链路中增加动态输出格式切换或重采样。

## 后续建议

1. 在 `case 80/81/82` 增加 `sd_file_count <= 0` 保护，避免空 SD 卡时访问空路径。
2. 为 SD 卡歌曲建立编号排序策略，避免 `readdir()` 返回顺序不稳定导致上一首/下一首体验不固定。
3. 统一 SD 卡文件名规范，优先使用短 ASCII 文件名。
4. 根据产品需求决定是要求离线转码，还是在端上加入重采样和声道转换。
