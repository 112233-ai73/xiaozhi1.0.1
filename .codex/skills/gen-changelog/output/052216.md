# OTA 与音频播放链路改动说明

## 背景

本次工作区相对 `HEAD` 共有 10 个文件修改，核心目标集中在三类能力：

1. 通过串口 `0x99` 指令触发小智固件 OTA。
2. OTA 前增加版本检查、网络状态日志、失败清理和语音识别任务恢复。
3. 调整 MP3 播放与语音识别之间的关系，减少开机提示音或播放提示音期间被 multinet 误识别的问题。

当前改动仍只覆盖 app 固件 OTA。`mp3` 分区由 `spiffs_create_partition_image(mp3 ./asset FLASH_IN_PROJECT)` 单独生成并烧录，未纳入 HTTP OTA 更新流程。

## 推荐方案：串口触发 + 云端版本检查 + 任务隔离

当前实现采用串口命令作为 OTA 入口：

```text
BB 04 99 26
```

协议层收到 `0x99` 后挂起语音识别相关任务，再创建 OTA 任务。OTA 任务启动后先读取本地 app 版本，再请求服务器版本 JSON；只有服务器版本高于本地版本时，才下载并写入新固件。

推荐继续沿用该方向，但需要补齐两个稳定性细节：

1. `stop_play_mp3()` 不应使用无延时无限 `while`，否则可能占满 CPU 或卡死调用任务。
2. OTA 版本 JSON 解析需要确认 `cJSON.h` 头文件和依赖已被正确引入，否则可能出现编译问题。

## 架构总览

| 模块 | 当前职责 | 本次变化 |
|---|---|---|
| 串口协议 | 校验并分发 `AA` / `BB` 命令 | `BB 0x99` 触发 OTA 前先挂起语音任务 |
| OTA | 下载并写入 app OTA 分区 | 增加本地/云端版本比较、失败清理、语音任务恢复 |
| 语音识别 | AFE/VAD/multinet 检测和命令队列 | 增加 multinet 开关，支持播放期间暂停识别 |
| MP3 播放 | 从 `mp3` SPIFFS 分区读取并异步播放 | 增加播放请求结构体，支持播放时暂停/恢复 multinet |
| 启动流程 | 初始化 WiFi、串口、音频和语音识别 | 恢复 WiFi 初始化和 OTA rollback 检查，启动提示音不参与 multinet |

## 详细设计

### 1. OTA 版本检查

`main/ota.c` 增加 `get_local_version()`，通过 `esp_ota_get_running_partition()` 和 `esp_ota_get_partition_description()` 读取当前运行固件版本。

`CMakeLists.txt` 增加：

```cmake
set(PROJECT_VER "1.0.0")
```

这会让固件描述信息中带有项目版本，供 OTA 版本比较使用。

云端版本地址定义在 `main/ota.h`：

```c
#define OTA_VERSION_URL "http://10.176.68.23:8000/xiaozhi_version.json"
```

预期 JSON 格式至少包含：

```json
{
  "version": "1.0.1"
}
```

版本比较由 `is_new_version_available()` 完成，目前按 `major.minor.patch` 三段数字比较。

### 2. OTA 失败清理与任务恢复

新增 `ota_fail_and_delete_task()`，统一处理失败路径：

| 清理动作 | 说明 |
|---|---|
| `http_cleanup(client)` | 关闭并释放 HTTP client |
| `esp_ota_abort(update_handle)` | 取消未完成的 OTA 写入 |
| `app_sr_resume_tasks()` | 恢复 OTA 前挂起的语音识别任务 |
| `vTaskDelete(NULL)` | 退出当前 OTA 任务 |

这比旧实现中各失败分支手动 cleanup 更集中，也避免 OTA 失败后语音任务长期停在挂起状态。

### 3. 串口命令触发 OTA

`main/app/app_usart_protocol.c` 中，`header == 0xBB` 且 `cmd_id == 0x99` 时增加：

```c
MY_LOGI("无关任务挂起");
app_sr_suspend_tasks();
MY_LOGI("OTA 开始");
start_xiaozhi_ota();
```

该路径的行为是：

```text
串口完整包 -> XOR 校验通过 -> BB 命令分发 -> 0x99 -> 挂起 SR 任务 -> 创建 OTA 任务
```

### 4. 语音识别任务挂起与恢复

`main/audio/audio_sr.c` 中 `app_sr_start()` 现在保存了 `sr_handler_task_handle`，使 `app_sr_suspend_tasks()` 可以同时挂起：

```text
audio_feed_task
audio_detect_task
sr_handler_task
```

恢复顺序为：

```text
sr_handler_task -> audio_detect_task -> audio_feed_task
```

这样 OTA 期间能减少音频采集、检测和命令处理对网络下载、Flash 写入的干扰。

### 5. 播放期间暂停 multinet

`audio_sr.c` 新增：

```c
void app_sr_set_multinet_enabled(bool enabled);
```

当关闭 multinet 时，会：

1. 设置 `s_multinet_enabled = false`。
2. 清空 `g_vad_speech`。
3. 调用 `clean_multinet()`。
4. 如果当前状态是 `LISTENING`，切回 `IDLE`。

`audio_detect_task()` 在每轮 fetch 后检查该开关：

```c
if (!s_multinet_enabled) {
    if (com_status == LISTENING) {
        switch_to_idle();
    }
    continue;
}
```

这意味着 AFE 仍然采集，但 multinet 命令识别被暂时跳过。

### 6. MP3 播放请求封装

`main/audio/audio_mp3_decode.c` 新增 `mp3_play_request_t`：

```c
typedef struct {
    char file_name[MP3_FILE_NAME_MAX_LEN];
    bool pause_multinet;
} mp3_play_request_t;
```

播放任务不再直接接收字符串指针，而是接收独立分配的请求对象。这样可以避免异步任务使用外部字符串生命周期不明确的问题，同时携带是否暂停 multinet 的配置。

新增两个公开播放接口：

| 接口 | 行为 |
|---|---|
| `audio_mp3_play_file_async()` | 正常异步播放，不暂停 multinet |
| `audio_mp3_play_file_async_without_multinet()` | 播放前关闭 multinet，播放结束或失败后恢复 |

`main/main.c` 的启动提示音已改为调用 `audio_mp3_play_file_async_without_multinet(STARTUP_MP3_FILE)`。

## 代码侧变化

| 文件 | 变化摘要 | 影响 |
|---|---|---|
| `CMakeLists.txt` | 增加 `PROJECT_VER` | 本地固件可提供版本号给 OTA 比较 |
| `main/ota.c` | 增加版本获取、云端 JSON 请求、版本比较、失败清理 | OTA 不再盲目升级，失败后恢复语音任务 |
| `main/ota.h` | 增加 `OTA_VERSION_URL` 和 `com_debug` 引入 | OTA 日志统一使用项目日志宏 |
| `main/app/app_usart_protocol.c` | `0x99` 触发 OTA 前挂起 SR 任务 | 串口 OTA 期间减少识别任务干扰 |
| `main/app/app_usart_protocol.h` | 引入 `audio/audio_sr.h` | 协议层可调用 SR 任务控制接口 |
| `main/audio/audio_sr.c` | 增加 multinet 开关，保存 handler task handle | 支持播放期间暂停识别和 OTA 期间挂起任务 |
| `main/audio/audio_sr.h` | 导出 `app_sr_set_multinet_enabled()` | MP3 模块可控制 multinet |
| `main/audio/audio_mp3_decode.c` | 播放请求对象化，增加 without_multinet 接口 | 提升异步播放参数安全性，并支持播放期间暂停识别 |
| `main/audio/audio_mp3_decode.h` | 导出 without_multinet 接口，隐藏 `mp3_player_task` | 外部不再直接创建播放器任务 |
| `main/main.c` | 恢复 WiFi 初始化和 rollback 检查，启动音改用 without_multinet | 开机进入可 OTA、可回滚验证的流程 |

## 当前方案映射

| 原方案 | 当前方案 | 行为变化 |
|---|---|---|
| 直接 HTTP 下载 `xiaozhi_update.bin` | 先请求 `xiaozhi_version.json`，确认版本更高才下载 | 避免重复升级同版本 |
| OTA 失败路径分散 cleanup | `ota_fail_and_delete_task()` 统一处理 | 降低资源泄漏和任务未恢复风险 |
| MP3 播放任务参数为字符串指针 | `mp3_play_request_t` 拷贝文件名 | 异步任务参数更稳定 |
| 播放期间 multinet 仍工作 | `audio_mp3_play_file_async_without_multinet()` 可暂停 multinet | 减少提示音被识别成命令的概率 |
| `sr_handler_task` 未保存句柄 | 创建任务时保存 `sr_handler_task_handle` | OTA 前可完整挂起 SR 链路 |

## 可删除或需要替换的代码

| 可删除/替换项 | 所在文件 | 行数 | 原因 |
|---|---|---:|---|
| `stop_play_mp3()` 中的无延时 `while (s_mp3_playing)` | `main/audio/audio_mp3_decode.c` | `~28` | 该循环会忙等，若播放器任务未及时退出，可能占满 CPU；建议替换为带 `vTaskDelay(pdMS_TO_TICKS(10))` 和超时的等待 |
| 外部直接声明/调用 `mp3_player_task()` 的方式 | `main/audio/audio_mp3_decode.h` | `~26` | 播放任务已内部化，外部应通过异步播放接口调用 |
| OTA 失败分支中分散的 cleanup 逻辑 | `main/ota.c` | 多处分支 | 已由 `ota_fail_and_delete_task()` 收敛，后续新增失败路径应复用统一函数 |

## 风险评估

| 风险点 | 触发条件 | 影响 | 建议验证 |
|---|---|---|---|
| `stop_play_mp3()` 忙等 | 正在播放 MP3 时调用停止，播放器任务未及时退出 | 调用任务可能卡住，系统调度受影响 | 改为 200ms 超时等待；连续唤醒打断播放测试 |
| OTA JSON 解析依赖缺失 | `ota.c` 使用 `cJSON` 但未显式包含 `cJSON.h` | 编译失败或隐式声明问题 | 执行完整 `idf.py build` |
| 版本字符串格式不符合 `x.y.z` | 云端返回 `1.0`、`v1.0.1` 或空字符串 | 版本比较结果不可靠 | 测试合法/非法 JSON 和多种版本格式 |
| 版本 HTTP client 初始化失败未判空 | `esp_http_client_init(&config_ver)` 返回 `NULL` | 后续 open/cleanup 可能异常 | 模拟内存不足或 URL 配置异常 |
| OTA 只更新 app 分区 | MP3 文件也发生变化 | OTA 后提示音仍是旧版本 | 如需资源升级，另做 `mp3` SPIFFS 分区 OTA |
| 挂起 SR 任务时 task handle 为空 | SR 初始化失败但仍触发 OTA | `vTaskSuspend(NULL)` 行为不可控 | 挂起前检查 task handle 或确认启动顺序 |

## 迁移步骤

1. 服务端准备两个文件：
   - `xiaozhi_version.json`
   - `xiaozhi_update.bin`
2. 确保 `xiaozhi_version.json` 至少包含：
   ```json
   {"version":"1.0.1"}
   ```
3. 固件构建时更新 `PROJECT_VER`，保证本地版本能和云端版本比较。
4. 设备启动后确认 WiFi 初始化成功，并执行 `check_ota_rollback()`。
5. 串口发送 `BB 04 99 26` 触发 OTA。
6. 观察日志中是否出现版本获取、版本比较、下载、写入、设置启动分区和重启信息。

## 验证方式

### OTA 验证

1. 本地版本低于云端版本时，确认设备下载并写入新固件。
2. 本地版本等于云端版本时，确认打印“无需升级”并恢复语音任务。
3. 关闭 HTTP 服务后触发 OTA，确认失败后 `app_sr_resume_tasks()` 被执行。
4. 使用无效固件文件，确认 `esp_ota_end()` 失败后不会切换启动分区。

### 音频与识别验证

1. 开机播放 `107.mp3` 时，确认 multinet 暂停，不会误触发命令。
2. 播放提示音期间连续讲话，确认状态不会反复 `LISTENING -> IDLE -> LISTENING` 抖动。
3. 唤醒词打断正在播放的 MP3，确认不会出现长时间 `MP3 playback busy`。
4. 播放结束或失败后，确认 `app_sr_set_multinet_enabled(true)` 被执行，语音识别恢复。

## 后续建议

1. 将 `stop_play_mp3()` 改成带超时的等待函数，例如最多等待 200ms，每 10ms 让出一次 CPU。
2. OTA 版本 JSON 建议增加 `url`、`sha256`、`mp3_version` 字段，为后续 MP3 分区 OTA 做准备。
3. 对 `mp3` 分区更新建议采用独立资源 OTA：下载 SPIFFS 镜像，校验后写入 `mp3` data 分区。
4. 如果用于量产，建议将单 `mp3` 分区改为 `mp3_0/mp3_1` 双资源分区，避免资源更新中断后不可恢复。
