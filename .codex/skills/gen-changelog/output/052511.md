# 在线语音上下行链路与音频缓冲改造说明

## 1. 背景

当前代码改动围绕“小智在线语音链路”展开，目标是在原有 SR/AFE 到 MultiNet 的离线命令词流程旁边，增加一条可供网络上传的音频数据通路，并补齐在线模式的下行播放链路。

原有主链路大致为：

```text
麦克风 -> audio_read -> AFE/SR -> MultiNet -> 离线命令词处理
```

本次改造后，在线语音链路建议按下面结构组织：

```text
上行:
麦克风 -> audio_read -> AFE fetch -> g_audio_out_ringbuf -> Opus 编码 -> websocket 发送

下行:
websocket 接收 -> Opus 解码 -> 播放 ringbuffer -> audio_write -> 扬声器
```

这里的关键点是：在送入 MultiNet 之前对 AFE 输出数据做一次拷贝，放入独立 ringbuffer。这样 MultiNet 仍然使用原始 `res->data` 指针进行命令词识别，而在线编码链路使用自己的缓冲区数据，避免后续异步编码或网络上传时访问到已经失效的 AFE 内部缓冲。

## 2. 推荐方案

建议把在线链路封装为 `audio_online_sr_handle`，让它只负责“音频数据桥接”，不要直接耦合 websocket 具体实现。

职责划分如下：

```text
audio_sr.c
- 负责从 AFE fetch 获取语音帧
- 在 MultiNet 识别前复制一份数据到上行 ringbuffer
- 暴露 app_sr_get_audio_ringbuf()

audio_encode.c
- 从上行 PCM ringbuffer 读取数据
- 编码为 Opus
- 输出到编码后 ringbuffer

audio_decode.c
- 从网络下行 ringbuffer 读取 Opus 数据
- 解码为 PCM
- 输出到播放 ringbuffer

audio_online_sr_handle.c
- 串起 encode/decode/ringbuffer/audio_write
- 对外提供 read_data/write_data，供 websocket 层调用

websocket 层
- 连接、鉴权、协议消息、发送编码数据、接收服务端音频数据
```

这样做的好处是 websocket 模块不需要知道音频采集、编码、解码和播放的内部细节；音频模块也不需要依赖具体网络协议。

## 4. 详细设计

### 4.1 SR 侧输出缓冲区

`audio_sr.c` 中新增的 `g_audio_out_ringbuf` 是在线上行的 PCM 缓冲区。它在 `app_sr_start()` 中创建，优先使用 PSRAM：

```text
AFE fetch 输出 -> 拷贝到 g_audio_out_ringbuf -> 继续交给 MultiNet
```

这里不能直接把 `res->data` 指针传给编码器，原因是 AFE fetch 返回的数据通常属于 AFE 内部缓冲，生命周期只在当前处理窗口可靠。编码器和网络发送是异步流程，如果只保存指针，后续读取时这块数据可能已经被 AFE 覆盖，导致上传音频错乱，同时也可能影响 MultiNet 实时识别。

当前拷贝策略是非阻塞写入 ringbuffer。如果 ringbuffer 满了，直接丢弃当前帧并打印 warning。这是合理的实时音频策略：优先保证 SR 和 MultiNet 不被网络或编码链路拖慢。

### 4.2 上行编码链路

在线 handle 启动后，会把 `app_sr_get_audio_ringbuf()` 返回的 PCM ringbuffer 注册给编码器：

```text
g_audio_out_ringbuf -> audio_encode -> encode_output_buffer
```

websocket 发送任务后续只需要调用：

```c
audio_online_sr_handle_read_data(handle, buffer, buffer_size, &read_size);
```

拿到的 `buffer` 内容应作为 websocket binary frame 发送。

### 4.3 下行解码链路

服务端下发的音频 binary frame 应写入在线 handle：

```c
audio_online_sr_handle_write_data(handle, payload, payload_len);
```

随后链路为：

```text
decode_input_buffer -> audio_decode -> decode_output_buffer -> audio_online_play_task -> audio_write
```

播放任务从 `decode_output_buffer` 中取 PCM，然后调用 `audio_write()` 输出到扬声器。

### 4.4 离线 SR 处理迁移

原 `main/app/app_sr_handler.c/.h` 已删除，离线命令词处理迁移到 `main/audio` 目录。这样音频处理相关代码集中在 audio 模块中，结构更清晰。

当前实现文件名为：

```text
main/audio/audio_offlline_sr_handler.c
```

而头文件名为：

```text
main/audio/audio_offline_sr_handler.h
```

建议后续把 `.c` 文件也改成 `audio_offline_sr_handler.c`，避免 `offlline/offline` 拼写不一致导致后续维护和 CMake 注册出错。

## 5. 需要注意的问题与改进建议

### 5.1 websocket 发送任务需要退出机制

当前 `audio_online_sr_handle_read_data()` 偏向阻塞式读取。如果 websocket 断开，而发送任务还在等待编码输出，任务退出会不够干净。

建议增加会话状态控制：

```text
connected = true  -> 允许 read + send
connected = false -> 停止发送循环，必要时 flush ringbuffer
```

也可以把读取等待从永久阻塞改成有限超时，让任务周期性检查连接状态。

### 5.2 编码输出 ringbuffer 满时不能长期卡住核心链路

如果网络发送速度慢于编码速度，编码输出 ringbuffer 会堆积。当前策略如果使用长时间阻塞，可能导致编码任务被卡住。

建议：

```text
网络断开或发送失败时，停止编码上传任务
编码输出满时允许丢帧
重连后清空旧数据，避免发送过期语音
```

实时语音上传更需要“新鲜数据”，不适合堆积太久。

### 5.3 AFE 输出格式要和编码器输入保持一致

需要确认以下参数一致：

```text
采样率: 16000 Hz
声道数: 建议 1 channel
采样格式: int16 PCM
帧长: 与 Opus encoder 配置一致
```

如果 AFE fetch 输出包含多通道数据，而编码器按单通道处理，需要在 SR 侧或编码器前做声道选择或重排。

### 5.4 PSRAM 分配失败需要完整清理

当前新增 ringbuffer、handle、编码器、解码器都依赖内存分配。建议所有 create/start 流程都补齐失败清理逻辑：

```text
任一步失败 -> 删除已创建 ringbuffer/task/codec -> 返回错误码
```

避免初始化失败后留下半初始化对象。

### 5.5 在线播放与离线识别可能互相干扰

服务端下发语音播放时，扬声器声音可能重新进入麦克风，导致本地 SR 或在线上传出现回声。

建议在服务端进入 speaking/TTS 状态时：

```text
暂停本地离线命令词检测，或降低上传策略
播放结束后再恢复 listening
```

如果后续有 AEC，则可以进一步优化。

## 6. websocket 成功连接后的处理流程

websocket 连接成功后，不建议立刻无控制地发送音频。建议按下面顺序处理。

### 6.1 设置连接状态

在 `WEBSOCKET_EVENT_CONNECTED` 中设置在线会话状态：

```text
online_connected = true
```

后续所有发送任务都先检查这个状态。

### 6.2 发送 hello/auth/session 配置

先发送小智协议要求的初始化消息。如果协议还没有最终定稿，至少应包含这些信息：

```json
{
  "type": "hello",
  "audio": {
    "format": "opus",
    "sample_rate": 16000,
    "channels": 1,
    "frame_duration": 60
  }
}
```

如果服务端需要 token、device_id、session_id，也应该在这个阶段发送。

### 6.3 创建并启动在线音频 handle

确保音频底层和 SR 已经启动：

```text
audio_init() 已完成
app_sr_start() 已完成
```

然后创建并启动：

```c
audio_online_sr_handle_t *handle = audio_online_sr_handle_create();
audio_online_sr_handle_start(handle);
```

如果 handle 已经启动，不要重复创建任务和 ringbuffer。

### 6.4 启动 websocket 上行发送任务

发送任务循环读取编码后的 Opus 数据：

```c
while (online_connected) {
    esp_err_t ret = audio_online_sr_handle_read_data(handle, tx_buf, tx_buf_size, &read_size);
    if (ret == ESP_OK && read_size > 0) {
        esp_websocket_client_send_bin(client, (const char *)tx_buf, read_size, portMAX_DELAY);
    }
}
```

发送失败时应退出循环，标记断开，并触发重连或恢复离线模式。

### 6.5 websocket 接收回调区分文本和二进制

接收事件中建议这样分流：

```text
文本 JSON:
- hello ack
- session 状态
- listening/speaking/stop
- 错误码

二进制数据:
- 认为是服务端下行音频
- 写入 audio_online_sr_handle_write_data()
```

二进制音频处理：

```c
audio_online_sr_handle_write_data(handle, data, data_len);
```

### 6.6 处理服务端 speaking/listening 状态

当服务端通知开始说话或开始下发 TTS 音频：

```text
进入 speaking
可暂停本地命令词处理或暂停上传
继续接收 binary 音频并播放
```

当服务端通知播放结束：

```text
进入 listening
恢复上传和本地 SR 策略
```

### 6.7 断开连接和重连处理

在 `WEBSOCKET_EVENT_DISCONNECTED` 或发送失败时：

```text
online_connected = false
停止或唤醒上行发送任务
清空编码输出和解码输入缓冲中的旧数据
停止当前在线播放或等待播放自然结束
恢复离线 SR/命令词能力
按策略重新连接 websocket
```

重连成功后重新发送 hello/session 配置，不建议继续发送断线前堆积的旧音频。

## 8. 后续建议

1. 将 `audio_offlline_sr_handler.c` 重命名为 `audio_offline_sr_handler.c`，同步修改 `CMakeLists.txt`。
2. 给 `audio_online_sr_handle` 增加 stop/destroy 接口，方便 websocket 断开时释放任务和缓冲。
3. websocket 发送任务使用有限超时读取，避免断线后无法及时退出。
4. 在下行播放期间加入状态机，明确 listening/speaking/idle 三种状态。
5. 统一音频帧参数，确认 AFE 输出声道数与 Opus 编码器输入声道数一致。
6. 对 ringbuffer 满、PSRAM 分配失败、websocket send 失败等路径补充日志和错误码。
