# BSP 在线通信与设备身份能力改动说明

## 1. 背景

本次代码改动围绕设备联网后的在线通信能力展开，新增了设备身份生成、NVS 持久化封装、HTTP 激活请求、WebSocket 会话通信以及动态响应缓存等 BSP 层模块。

从已有上下文看，当前工程正在讨论离线语音资源迁移、设备是否具备联网能力的 `is_online` 标志、唤醒后 WebSocket 连接生命周期，以及唤醒词是否需要同步给后端的问题。本次代码改动正好为“唤醒后建立在线通道、上传音频、接收后端音频或文本回复”提供了基础模块。

需要注意的是，当前新增文件已经加入 `main/CMakeLists.txt` 编译，但从全工程搜索结果看，这些新接口目前主要停留在 BSP 模块层，尚未看到业务入口主动调用 `bsp_ws_init()`、`bsp_ws_start()`、`http_request_active_code()` 或 `bsp_NVS_Init()`。

## 2. 当前改动概览

| 文件 | 改动类型 | 说明 |
| --- | --- | --- |
| `main/CMakeLists.txt` | 修改 | 将 `bsp_esp32s3.c`、`bsp_http.c`、`bsp_mutable_buffer.c`、`bsp_nvs.c`、`bsp_ws.c` 加入编译源文件列表 |
| `main/bsp/bsp_esp32s3.c/.h` | 新增 | 提供 MAC 地址获取、UUID 生成与读取能力 |

| `main/bsp/bsp_mutable_buffer.c/.h` | 新增 | 提供可扩展响应缓存，用于 HTTP 接收分片数据 |

| `main/bsp/bsp_ws.c/.h` | 新增 | 提供 WebSocket 初始化、连接、关闭、hello、listen、abort、opus、IoT 状态上报等接口 |

## 3. 推荐整体设计

建议把本次新增能力定位为“在线通道 BSP 层”，不要直接把业务状态散落在各个发送函数中。

推荐链路如下：

```text
系统启动
  -> NVS 初始化
  -> WiFi 初始化与联网
  -> 获取 MAC / UUID
  -> 可选 HTTP 激活 / OTA 检查

唤醒成功
  -> 异步启动 WebSocket
  -> 播放本地唤醒反馈音
  -> WebSocket 连接成功后发送 hello
  -> 获取 session_id
  -> 发送 listen start / wakeup detect / opus 音频

退出唤醒或监听结束
  -> 发送 listen stop 或 abort
  -> 关闭 WebSocket
  -> 设备状态回到 IDLE
```

这个设计与当前“唤醒时连接 WebSocket，退出唤醒时断开 WebSocket”的思路是匹配的。两秒唤醒音频播放期间使用异步任务连接 WebSocket，也能避免阻塞后续音频上传。

## 4. 详细设计说明

### 4.1 设备身份模块

`bsp_esp32s3.c` 新增两个核心接口：

- `bsp_get_mac()`：通过 `esp_wifi_get_mac(WIFI_IF_STA, ...)` 获取 STA MAC，并格式化为 `xx:xx:xx:xx:xx:xx`。
- `bsp_get_uuid()`：优先从 NVS 读取 `UUID`，不存在时生成一个 UUID 字符串并写入 NVS。

该模块用于 HTTP 和 WebSocket Header：

- `Device-Id`
- `Client-Id`

当前设计方向是合理的：MAC 作为设备硬件身份，UUID 作为客户端持久身份。

### 4.2 NVS 封装模块

`bsp_nvs.c` 新增 `storage` 命名空间，并封装：

- `bsp_NVS_Init()`
- `bsp_NVS_WriteStr()`
- `bsp_NVS_ReadStr()`
- `bsp_NVS_DeleteKey()`
- `bsp_NVS_FindKey()`
- `bsp_NVS_DeleteAllKey()`

该模块主要服务于 UUID 持久化。后续如果要保存 `is_online`、激活码、OTA 资源版本号、离线语音资源版本号，也可以复用该模块。

### 4.3 HTTP 激活请求模块

`bsp_http.c` 新增 `http_request_active_code()`，向：

```text
https://api.tenclass.net/xiaozhi/ota/
```

发起 HTTPS POST 请求。

请求体通过 `protocol_http_get_body()` 生成，包含：

- `application.version`
- `application.elf_sha256`
- `board.type`
- `board.name`
- `board.ssid`

请求 Header 包含：

- `Device-Id`
- `Client-Id`
- `User-Agent`
- `Accept-Language`

响应数据通过 `bsp_mutable_buffer` 追加缓存，适合处理 HTTP 分片接收。

### 4.4 WebSocket 在线通道模块

`bsp_ws.c` 新增 WebSocket 客户端，目标地址为：

```text
wss://api.tenclass.net/xiaozhi/v1/
```

主要接口包括：

- `bsp_ws_init()`：初始化客户端、注册事件、添加认证与设备 Header。
- `bsp_ws_start()`：启动连接并等待连接成功事件位。
- `bsp_ws_stop()`：关闭 WebSocket。
- `bsp_ws_send_hello()`：发送 hello，并等待 session_id 事件位。
- `bsp_ws_send_wakeup_word()`：发送唤醒词 detect 消息。
- `bsp_ws_send_start_listen()`：发送监听开始。
- `bsp_ws_send_stop_listen()`：发送监听结束。
- `bsp_ws_send_abort()`：发送中断请求。
- `bsp_ws_send_opus()`：发送 Opus 二进制音频帧。
- `bsp_ws_send_iot_learn()` / `bsp_ws_send_iot_status()`：发送 IoT 描述和状态。

该模块已经具备在线语音链路的基本发送能力，后续重点是和音频采集、唤醒状态机、后端回复播放链路进行串接。

## 5. 当前方案与前序设计问题的对应关系

### 5.1 关于 `is_online`

当前代码还没有新增 `is_online` 标志位。建议后续放在 WiFi/IP 事件回调处维护：

- 获取 IP 成功：`is_online = 1`
- WiFi 断开、IP 丢失、连接失败超时：`is_online = 0`

HTTP 和 WebSocket 层只消费该状态，不建议由它们反向决定设备是否在线。

### 5.2 关于唤醒后是否发送唤醒词给后端

当前 `bsp_ws_send_wakeup_word()` 明确会向后端发送：

```json
{
  "type": "listen",
  "state": "detect",
  "text": "你好小智"
}
```

这说明当前代码倾向于“本地 Multinet 唤醒后，仍向后端同步一次唤醒事件”。如果后端不应该对唤醒词本身回复语音，需要后端根据 `state=detect` 或协议字段过滤，不返回业务语音回复。

### 5.3 关于连接即代表唤醒

当前 WebSocket 连接生命周期可以设计为：

- 唤醒：启动 WebSocket
- 监听中：保持 WebSocket
- 退出唤醒：关闭 WebSocket

但协议上仍建议保留 `hello`、`listen start`、`listen stop`、`abort` 这类显式状态消息。连接状态只能说明通道存在，不能完整表达“正在监听、检测到唤醒词、用户停止说话、被新唤醒打断”等语义。

## 6. 当前需要重点修正的风险

1. `bsp_NVS_Init()` 尚未发现调用入口  
   `bsp_get_uuid()` 依赖 NVS 句柄 `my_handle`，如果没有先调用 `bsp_NVS_Init()`，读写 UUID 会存在失败风险。

2. NVS 写入和删除后缺少 `nvs_commit()`  
   `bsp_NVS_WriteStr()`、`bsp_NVS_DeleteKey()`、`bsp_NVS_DeleteAllKey()` 当前只调用 set/erase，没有提交，断电后可能无法持久保存。

3. MAC/UUID 字符串存在释放责任不清晰的问题  
   `bsp_get_mac()` 和 `bsp_get_uuid()` 会动态分配内存，但 HTTP/WS 设置 Header 后没有看到释放逻辑，存在内存泄漏风险。

4. WebSocket 等待可能永久阻塞  
   `bsp_ws_start()` 和 `bsp_ws_send_hello()` 使用 `portMAX_DELAY` 等待事件位。网络异常、后端无响应或 session_id 未解析时，任务可能卡死。

5. `session_id` 事件位依赖外部回调设置  
   当前 `bsp_ws.c` 内部没有解析 hello 响应并设置 `WEBSOCKET_SESSION_ID_BIT`，需要确认 `text_callback` 中是否会写入 `session_id` 并设置事件位。

6. WebSocket 认证 Token 仍是测试值  
   `WS_TOKEN` 当前为 `test-token`，后续应接入真实 token 获取、刷新或配置逻辑。

7. HTTP 资源释放不完整  
   `protocol_http_get_body()` 返回的 body、`esp_http_client_init()` 创建的 client、以及动态生成的 Header 字符串需要明确释放。

8. 动态缓存缺少申请失败保护  
   `bsp_mutable_buffer_create()`、`heap_caps_malloc()`、`heap_caps_realloc()` 当前没有判空，内存紧张时可能崩溃。

9. 新增文件仍是未跟踪状态  
   当前 `git status` 显示新增 BSP 文件均为 untracked，需要在提交前确认是否全部纳入版本控制。

## 7. 建议后续迁移步骤

1. 在系统初始化阶段明确调用 `bsp_NVS_Init()`，并保证早于 `bsp_get_uuid()`。
2. 修正 NVS 写入、删除后的 `nvs_commit()`。
3. 将 `bsp_get_mac()`、`bsp_get_uuid()` 的内存所有权固定下来，避免 Header 设置后泄漏。
4. 为 WebSocket 连接、hello、session_id 等等待增加超时和失败返回值。
5. 在 `text_callback` 中统一解析后端 JSON，处理 `hello/session_id`、TTS、IoT、错误码等消息。
6. 将 `WS_TOKEN`、服务端 URL、语言、设备型号等配置项收敛到配置文件或 NVS。
7. 把 `is_online` 放到 WiFi/IP 事件层维护，WebSocket 和 HTTP 根据该状态决定是否发起请求。
8. 如果后续要通过 OTA 更新 W25Q128 中的离线语音资源，建议复用 HTTP/OTA 检查能力，新增资源版本号、资源分区表、下载校验、写入 Flash 和回滚保护。

## 8. 验证建议

建议至少完成以下验证：

- 编译验证：确认新增 BSP 文件加入 CMake 后无编译错误。
- NVS 验证：首次启动生成 UUID，重启后读取同一个 UUID。
- HTTP 验证：确认激活请求能拿到完整响应，且多次请求无明显堆内存下降。
- WebSocket 验证：连接、hello、session_id、listen start、opus 上传、listen stop、close 全链路可运行。
- 异常验证：断网、DNS 失败、后端无响应、WebSocket 被动断开时，任务不会永久阻塞。
- 唤醒链路验证：本地唤醒反馈音播放期间异步连接 WebSocket，不影响后续音频采集与上传。

## 9. 结论

本次改动已经把在线语音通信所需的基础 BSP 能力搭起来了，方向上与“唤醒后连接 WebSocket，退出唤醒后断开”的方案一致。

当前最需要补齐的是初始化入口、资源释放、NVS commit、WebSocket 超时机制和 session_id 回调解析。完成这些后，这套模块就可以比较稳定地承载后续在线语音、后端回复播放，以及离线语音资源 OTA 到外部 Flash 的能力。
