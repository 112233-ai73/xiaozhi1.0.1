# 在线回复可被 0/1/2 命令打断改动说明

## 背景

本次改动围绕“设备正在在线回复或播放提示音时，仍允许命令 `0/1/2` 打断当前 `SPEAKING` 状态”展开。

历史说明中最近的语音链路状态为：

```text
IDLE -> LISTENING -> SPEAKING -> IDLE
```

其中 `audio_sr.c` 已经改为由 AFE VAD 驱动进入 `LISTENING`，再由 MultiNet 识别离线命令；MP3 播放完成后会回到 `IDLE`。本轮问题出现在 `SPEAKING` 阶段：在线 TTS 或本地 MP3 正在播放时，原逻辑会抑制识别，或者识别后走到未唤醒分支，导致不能按预期执行唤醒态下的 `0/1/2` 打断流程。

当前工作区相对 `HEAD` 修改 6 个文件：

| 文件 | 变更规模 | 主要内容 |
|---|---:|---|
| `main/audio/audio_mp3_decode.c` | `~11 行` | MP3 异步播放创建成功后立即恢复唤醒态，并在播放结束/停止/错误时清理 `MP3_after_awake` |
| `main/audio/audio_offline_sr_handler.c` | `~7 行` | 唤醒态 `0/1/2` 命令增加在线播放打断，退出命令增加 WebSocket 停止 |
| `main/audio/audio_offline_sr_handler.h` | `~1 行` | 引入 `bsp_ws.h`，用于调用 `bsp_ws_stop()` |
| `main/audio/audio_online_sr_handle.c` | `~54 行` | 增加在线 TTS 打断标志、解码缓冲清空、二进制音频丢弃、abort 发送 |
| `main/audio/audio_online_sr_handle.h` | `~1 行` | 对外暴露 `audio_online_interrupt_playback()` |
| `main/audio/audio_sr.c` | `~21 行` | 允许 `SPEAKING` 状态下识别 `0/1/2`，过滤其他命令，并限制在线音频上传条件 |

## 推荐方案：用专用 interrupt 标志截断在线 TTS 播放链路

本次方案不直接销毁在线播放任务，而是新增一个轻量的播放中断标志 `s_interrupt_online_playback`，让在线音频链路的三个入口都感知“当前回复已经被打断”：

1. WebSocket 收到新的二进制 TTS 音频时直接丢弃。
2. 已经解码到播放缓冲的数据不再写入 codec。
3. 打断瞬间主动清空 decode 输入和输出 ringbuffer，并向服务端发送 `abort`。

这样可以避免旧回复残留音频继续播放，也能避免服务端延迟到达的 `tts start` 把已经打断的旧回复重新拉回 `SPEAKING`。

## 架构总览

### 在线回复打断链路

```text
SPEAKING
  -> AFE VAD 检测到人声
  -> MultiNet 继续识别
  -> 只允许 command_id 0/1/2 通过
  -> sr_handler_task 收到 0/1/2
  -> stop_play_mp3()
  -> audio_online_interrupt_playback()
       -> s_interrupt_online_playback = true
       -> 清空 decode_input_buffer / decode_output_buffer
       -> bsp_ws_send_abort()
       -> is_awake 时切到 IDLE
  -> 播放 /spiffs/104.mp3
```

### 在线 TTS 接收与播放链路

```text
WS binary
  -> audio_online_ws_bin_cb()
       -> s_interrupt_online_playback 为 true 时丢弃
  -> decode_input_buffer
  -> audio_decode
  -> decode_output_buffer
  -> audio_online_play_task()
       -> s_interrupt_online_playback 为 false 才 audio_write()
```

### 状态流转

```text
在线正常回复:
LISTENING -> tts start -> SPEAKING -> tts stop -> IDLE

在线回复被 0/1/2 打断:
SPEAKING -> VAD speech -> MultiNet detected 0/1/2 -> IDLE -> 播放 104.mp3

本地 MP3 提示音期间:
SPEAKING -> VAD speech -> MultiNet detected 0/1/2 -> stop_play_mp3() -> 播放 104.mp3
```

## 详细设计

### MP3 播放唤醒态恢复

`main/audio/audio_mp3_decode.c` 中，`audio_mp3_play_file_async_internal()` 在成功创建 `mp3_player_task` 后，如果 `MP3_after_awake` 为 true，会立即执行：

```c
com_set_awake(true);
MP3_after_awake = false;
```

旧逻辑把这段代码注释掉，导致本地 MP3 播放过程中 `is_awake` 仍可能保持 false。此时说 `0/1/2` 会进入未唤醒分支，表现为执行“未唤醒状态下的 `0/1/2` 指令”。新逻辑在播放任务创建成功后立即恢复唤醒态，让播放期间的打断命令进入唤醒态分支。

同时，`mp3_player_task()` 在以下三类出口都会清理 `MP3_after_awake`：

| 出口 | 行为 |
|---|---|
| `s_mp3_stop_requested` | 打印 stopped，切 `IDLE`，清理 `MP3_after_awake` |
| 解码错误 | 打印错误，清理 `MP3_after_awake` |
| 正常播放完成 | 打印 finished，切 `IDLE`，清理 `MP3_after_awake` |

这能避免一次播放遗留的唤醒恢复标志影响下一次播放。

### 离线命令 handler 增加在线播放打断

`main/audio/audio_offline_sr_handler.c` 的唤醒态命令 `0/1/2` 分支现在执行：

```text
stop_play_mp3()
audio_online_interrupt_playback()
com_set_awake(true)
audio_mp3_play_file_async("/spiffs/104.mp3")
```

这表示 `0/1/2` 不只打断本地 MP3，也会打断在线 TTS。这样在线回复、离线提示音两种播放来源都能被同一组命令中断。

命令 `3` 额外增加 `bsp_ws_stop()`，用于退出唤醒或结束在线会话时关闭 WebSocket 通道。

### 在线播放 interrupt 标志

`main/audio/audio_online_sr_handle.c` 新增：

```c
static volatile bool s_interrupt_online_playback = false;
```

该标志的职责是区分“正常在线回复播放”和“已经被本地命令打断的旧回复”。它被以下位置使用：

| 位置 | 行为 |
|---|---|
| `audio_online_ws_bin_cb()` | 标志为 true 时，新的 TTS 二进制音频直接丢弃 |
| `audio_online_play_task()` | 标志为 true 时，decode 输出数据不再 `audio_write()` |
| `handle_tts_message()` | 如果已打断且当前不是 `LISTENING`，延迟到达的 `tts start` 直接忽略 |
| `audio_online_interrupt_playback()` | 设置标志、清空缓冲、发送 abort、状态回到 `IDLE` |

其中 `handle_tts_message()` 的保护比较关键。在线回复被打断后，服务端可能还有滞后的 `tts start` 或音频包到达。如果在任意 `tts start` 时都清掉 interrupt 标志，就可能让旧回复重新播放。当前逻辑只允许在设备已经重新进入 `LISTENING` 时，新的在线会话清除 interrupt 标志。

### 清空在线解码缓冲

新增 `drain_ringbuffer()`，用于把 ringbuffer 中已经排队但还没播放的数据取出并归还：

```text
decode_input_buffer  -> 待解码的服务端 Opus 数据
decode_output_buffer -> 已解码待播放的 PCM 数据
```

`audio_online_interrupt_playback()` 会同时清空这两个缓冲，避免出现以下残留：

1. 已收到但尚未解码的旧 TTS 音频。
2. 已解码但尚未写入 codec 的旧 PCM。

### SPEAKING 状态下的 VAD 与 MultiNet 策略

`main/audio/audio_sr.c` 中 `handle_vad_state()` 原先在 `SPEAKING` 状态直接返回 false：

```text
SPEAKING -> 不识别 -> 无法打断
```

现在改为：

```text
SPEAKING -> is_awake && vad_speech 时继续 MultiNet 检测
```

这样设备播放在线回复时，用户说话仍然可以进入离线命令识别。但为了避免任意离线命令都打断播放，`handle_multinet_detect()` 增加过滤：

```text
SPEAKING 状态下:
  command_id == 0/1/2 -> 发送给 sr_handler_task
  其他 command_id -> clean_multinet() 后忽略
```

同时，`ESP_MN_STATE_TIMEOUT` 在 `SPEAKING` 状态只清理 MultiNet，不再发送 timeout 结果和切状态，避免播放期间因为识别超时把状态机拉回 `IDLE`。

### 在线音频上传条件收紧

`audio_detect_task()` 中给在线识别 ringbuffer 发送音频的条件从：

```text
com_status == LISTENING && is_wsline == true
```

收紧为：

```text
com_status == LISTENING && is_awake == true && is_wsline == true
```

这样可以避免未唤醒状态下因为 VAD 误触发进入 `LISTENING` 后仍向在线服务上传音频。

### AFE 配置调整

`configure_afe()` 中：

```c
afe_config->se_init = false;
```

关闭了双麦声源增强初始化。结合前面日志中出现过的 AFE feed ringbuffer full 风险，当前配置更偏向减少 AFE 处理负载，让 `fetch()` 更稳定地消费前端数据。

## 与当前方案的映射

| 旧行为 | 新行为 | 说明 |
|---|---|---|
| `SPEAKING` 状态直接跳过识别 | `SPEAKING` 且唤醒、有语音时继续识别 | 支持在线回复或提示音播放中被 `0/1/2` 打断 |
| 播放期间 `is_awake` 可能保持 false | MP3 task 创建成功后立即 `com_set_awake(true)` | `0/1/2` 进入唤醒态命令分支 |
| `stop_play_mp3()` 只能处理本地 MP3 | `stop_play_mp3()` + `audio_online_interrupt_playback()` | 本地播放和在线 TTS 都能被打断 |
| 在线旧音频包仍可能到达并播放 | interrupt 标志阻止 bin 接收和 audio_write | 避免旧回复残留播放 |
| TTS `start` 可能清掉打断状态 | 非 `LISTENING` 下的旧 `tts start` 被忽略 | 避免旧回复复活 |
| 未唤醒但 WebSocket 在线时可能上传音频 | 上传前要求 `is_awake == true` | 更符合唤醒后在线交互语义 |

## 风险评估

| 风险点 | 触发条件 | 影响 | 建议验证 |
|---|---|---|---|
| 播放回声触发 VAD | 在线 TTS 或 MP3 外放音量较大 | `SPEAKING` 下会进入 MultiNet，可能增加 CPU 占用或误识别 | 在线回复时不说话，观察是否频繁出现 `Deteted command` |
| `s_interrupt_online_playback` 长时间保持 true | 打断后没有进入新的 `LISTENING` 或服务端状态不同步 | 后续 TTS 二进制包被持续丢弃 | 打断后重新唤醒并发起在线问答，确认新回复能播放 |
| `bsp_ws_send_abort()` 与服务端状态不一致 | WebSocket 已断开或服务端未正确处理 abort | 服务端仍可能继续推送旧音频 | 观察打断后是否还有二进制包到达，确认本地会丢弃 |
| 命令过滤只允许 `0/1/2` | 播放期间用户说其他床控命令 | 命令被忽略，不会执行床控 | 这是当前策略设计，需要确认产品交互是否只允许停止/打断类命令 |
| 关闭 `se_init` 改变识别效果 | 双麦环境依赖声源增强 | 可能影响部分命令识别率 | 对 `0/1/2` 和常用命令做安静/播放中两类测试 |
| `MP3_after_awake` 提前清理 | 某些提示音仍希望播放完成后才恢复唤醒 | 行为从播放后恢复变为播放开始后恢复 | 验证启动提示音、唤醒提示音、命令反馈音的状态流转 |

## 迁移步骤

1. 保留 `audio_online_start_async()` 的在线唤醒入口不变。
2. 在离线命令 handler 中使用 `audio_online_interrupt_playback()` 作为在线 TTS 的统一打断接口。
3. 在 `SPEAKING` 状态下仅放行 `command_id` 为 `0/1/2` 的 MultiNet 结果。
4. 保持 `stop_play_mp3()` 处理本地 MP3，`audio_online_interrupt_playback()` 处理在线 TTS，两者在 `0/1/2` 分支同时调用。
5. 线上回复被打断后，依赖下一轮 `LISTENING` 和新的 TTS `start` 清除 interrupt 状态。

| AFE 实时性 | 连续播放和打断多次 | 不应大量出现 `Ringbuffer of AFE(FEED) is full` |

## 当前状态结论

当前改动已经把“在线回复可被 `0/1/2` 打断”拆成两层处理：

1. `audio_sr.c` 负责在 `SPEAKING` 状态下继续识别，并只放行 `0/1/2`。
2. `audio_online_sr_handle.c` 和 `audio_offline_sr_handler.c` 负责真正停止旧播放、清空在线音频缓冲、发送 abort，并播放本地打断提示音。

这套方案的核心收益是打断路径比较集中，不需要重启 WebSocket 或销毁在线任务。主要风险在于播放回声导致 VAD/MultiNet 额外工作，以及 interrupt 标志和服务端 TTS 状态的时序，需要通过板端连续打断测试确认。
