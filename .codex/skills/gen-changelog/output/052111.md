# USART 与语音处理模块分层重构说明

## 背景

本次改动围绕 ESP-IDF 主组件的目录职责重新划分展开，重点是将串口驱动、串口协议解析、语音识别结果处理从原先较混杂的位置拆到更明确的层级中。

重构前，`main/usart/usart_init.c` 同时承担 UART 驱动初始化、数据包接收、校验、WiFi 配网分片处理、普通命令处理等职责；`main/audio/audio_sr_handler.c` 则位于 `audio` 目录下，但实际负责语音识别结果到业务控制指令的映射。重构后，代码开始向以下结构迁移：

| 层级 | 目标职责 |
|---|---|
| `bsp` | 板级外设驱动，例如 UART 初始化、收发、事件任务 |
| `app` | 应用层业务逻辑，例如串口协议解析、语音识别命令处理 |
| `audio` | 音频初始化、语音识别模型、MP3 播放等音频能力 |

## 推荐方案：驱动层与协议层拆分

当前方案建议继续保持 `bsp_usart` 只负责 UART 物理通信和数据包完整性检查，协议语义全部交给 `app_usart_protocol`。这样可以减少 BSP 层对业务模块的依赖，让串口驱动后续更容易复用或替换。

## 架构总览

当前工作区中的主要迁移关系如下：

| 原位置 | 新位置 | 说明 |
|---|---|---|
| `main/usart/usart_init.c` | `main/bsp/bsp_usart.c` | UART 初始化、发送、接收任务保留在 BSP 层 |
| `main/usart/usart_init.h` | `main/bsp/bsp_usart.h` | 对外暴露 `usart_init()` 和 `usart_send_data()` |
| `main/audio/audio_sr_handler.c` | `main/app/app_sr_handler.c` | 语音识别结果处理迁移到应用层 |
| `main/audio/audio_sr_handler.h` | `main/app/app_sr_handler.h` | `sr_handler_task()` 声明迁移到应用层头文件 |
| `usart_init.c` 内部协议处理 | `main/app/app_usart_protocol.c` | WiFi 分片、串口命令分发迁移到应用层 |

新的串口数据流为：

1. `usart_init()` 初始化 UART 驱动、事件队列和 `usart_event_task`。
2. UART 收到数据后由 `usart_receive_callback()` 读取完整包。
3. `usart_packet_checksum_passed()` 完成 XOR 校验。
4. `usart_dispatch_packet()` 记录日志后调用 `app_usart_protocol_handle_packet()`。
5. `app_usart_protocol_handle_packet()` 按命令类型分发到 WiFi 配网分片处理或普通命令处理。

## 详细设计

### BSP USART 驱动层

`main/bsp/bsp_usart.c` 保留底层串口能力：

| 功能 | 当前实现 |
|---|---|
| UART 参数配置 | `UART_NUM_0`、`GPIO_NUM_43`、`GPIO_NUM_44`、`115200` |
| 发送接口 | `usart_send_data(const uint8_t *data)` |
| 接收任务 | `usart_event_task()` 监听 `UART_DATA`、`UART_FIFO_OVF`、`UART_BUFFER_FULL` |
| 包读取 | `usart_read_packet()` 读取帧头、长度和剩余数据 |
| 校验 | `calculate_xor_checksum()` + `usart_packet_checksum_passed()` |
| 协议交接 | `app_usart_protocol_handle_packet(packet, packet_len)` |

与旧实现相比，BSP 层不再直接处理 WiFi SSID/password 分片，也不再直接播放 200 到 206 的提示音；这些业务语义已迁到应用层。

### App USART 协议层

`main/app/app_usart_protocol.c` 新增了应用层协议解析：

| 功能 | 当前实现 |
|---|---|
| WiFi 分片识别 | `WIFI_CMD_SSID`、`WIFI_CMD_PASSWORD` |
| 分片缓存 | `wifi_fragment_context_t` 保存 SSID、password 和 `packet_mask` |
| 分片完成判断 | `WIFI_FRAGMENT_COMPLETE_MASK` |
| 配网触发 | `bsp_wifi_start_connect()` |
| 普通命令处理 | `process_normal_command()` |
| MP3 命令映射 | `0x6A ~ 0x76` 偶数命令映射到 `200.mp3 ~ 206.mp3` |

这一层开始承接旧 `usart_init.c` 内部的协议职责。当前普通命令处理使用数学规律替代原先多个 `case` 分支，减少重复代码。

### App SR Handler 迁移

`main/app/app_sr_handler.c` 从原 `audio` 目录迁出，职责没有明显变化，仍然处理 `sr_result_t` 队列数据，并根据 `command_id` 执行：

| 行为 | 相关接口 |
|---|---|
| 唤醒状态切换 | `com_set_awake(true)` |
| 状态流转 | `com_status_change(SPEAKING/IDLE)` |
| 语音反馈 | `audio_mp3_play_file_async()` |
| 串口控制发送 | `usart_send_data()` |

该模块仍由 `audio_sr.c` 创建任务：

```c
xTaskCreatePinnedToCore(sr_handler_task, "SR Handler Task", ...);
```

因此迁移后需要保证 `audio_sr.h` 能正确包含 `app/app_sr_handler.h`。

## 当前方案映射

| 当前实现 | 新方案 | 说明 |
|---|---|---|
| `usart_init()` | `bsp_usart.c::usart_init()` | 接口名保持不变，调用方改 include 即可 |
| `usart_send_data()` | `bsp_usart.c::usart_send_data()` | 语音命令发送逻辑可继续使用 |
| UART 接收后直接处理业务命令 | UART 接收后转交 `app_usart_protocol_handle_packet()` | BSP 不再感知具体业务命令 |
| `FRAME_HEADER_CMD 0xAA` | `FRAME_HEADER_A 0xAA` + `FRAME_HEADER_B 0xBB` | 新实现允许两类帧头，需要确认协议设计是否预期如此 |
| WiFi 分片处理内聚在 USART 文件 | `app_usart_protocol.c` 单独维护分片状态 | 配网协议逻辑更靠近应用层 |
| `audio/audio_sr_handler.*` | `app/app_sr_handler.*` | 语音识别结果处理被归类为应用逻辑 |

## 风险评估

| 风险点 | 触发条件 | 影响 | 建议验证 |
|---|---|---|---|
| CMake 源文件路径不一致 | `SRCS` 中路径与实际文件名不一致 | CMake 生成失败或目标无源文件 | 执行 `idf.py build`，重点检查 `main/CMakeLists.txt` 中 `app/app_sr_handler.c` 是否存在 |
| 新增文件未纳入 Git | `main/app/*`、`main/bsp/bsp_usart.*` 仍为 untracked | 提交后其他环境缺文件 | 提交前检查 `git status --short` |
| `FRAME_HEADER_B` 行为变化 | 外设只期望 `0xAA` 帧头，但固件也接受 `0xBB` | 可能接收非预期数据包 | 用串口工具分别发送 `0xAA`、`0xBB` 帧验证 |
| 协议层头文件依赖较重 | `app_usart_protocol.h` 包含 WiFi、MP3、debug 头文件 | 头文件传播业务依赖，后续编译耦合增加 | 可考虑将业务 include 下沉到 `.c` 文件 |
| 普通命令映射变更 | 原 `case 0x6A~0x76` 被数学映射替代 | 若未来存在不连续命令，可能误映射 | 对 `0x6A` 到 `0x76` 做逐项回归验证 |
| 注释编码异常 | 文件中中文注释显示为乱码 | 不影响逻辑，但影响维护 | 统一源码编码为 UTF-8 |

## 迁移步骤

1. 保留 `usart_init()`、`usart_send_data()` 这两个外部接口名，降低调用方迁移成本。
2. 将调用方 include 从 `usart/usart_init.h` 切换到 `bsp/bsp_usart.h`。
3. 将 `audio_sr_handler` 的 include 从 `audio_sr_handler.h` 切换到 `app/app_sr_handler.h`。
4. 在 `main/CMakeLists.txt` 中确保 `app/app_sr_handler.c`、`app/app_usart_protocol.c`、`bsp/bsp_usart.c` 都在 `SRCS` 中。
5. 删除旧目录 `main/usart` 下的 `usart_init.c/.h`，删除旧 `main/audio/audio_sr_handler.c/.h`。
6. 执行 `idf.py fullclean` 后重新 `idf.py build`，避免旧构建缓存影响判断。
7. 通过串口工具验证命令收发、WiFi 分片配网、语音命令控制三条主路径。

## 验证方式

建议按以下顺序验证：

1. 编译验证：执行 `idf.py build`，确认 `main` 组件源文件路径全部正确。
2. 启动验证：烧录后确认 `app_main()` 能依次执行 `bsp_wifi_init()`、`usart_init()`、`audio_init()`、`app_sr_start()`。
3. 串口收包验证：发送合法 XOR 校验包，确认日志出现 `checksum passed`。
4. 协议分发验证：发送 `0x00/0x02` WiFi 分片，确认最终调用 `bsp_wifi_start_connect()`。
5. 普通命令验证：发送 `0x6A`、`0x6C`、`0x76`，确认分别播放 `200.mp3`、`201.mp3`、`206.mp3`。
6. 语音控制验证：触发唤醒和床体控制命令，确认 `sr_handler_task()` 仍能通过 `usart_send_data()` 输出控制帧。

## 当前状态结论

当前改动方向是合理的：目录职责更清晰，串口驱动与协议解析开始解耦，语音识别 handler 也从音频底层目录迁移到应用层。提交前需要重点确认 CMake 文件路径、未跟踪文件纳入版本控制，以及串口协议帧头变化是否符合实际设备协议。
