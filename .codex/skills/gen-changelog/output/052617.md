# 当前代码改动说明

## 背景

本次工作区相对 `HEAD` 的改动主要集中在四类问题：

1. SD 卡 MP3 播放音量偏低，针对重采样后的 PCM 增加软件增益。
2. 语音识别状态机要求 `LISTENING` 只能在 `is_awake == true` 时进入，但未唤醒时仍允许 MultiNet 消费人声数据用于唤醒/命令检测链路。
3. HTTP 激活码请求和可变缓冲区补充错误处理、资源释放和 PSRAM 分配检查。
4. 为后续 WiFi 连接等待、WebSocket 启动和激活码请求预留入口，但 `main.c` 中相关调用目前仍保持注释状态。

当前工作区涉及 10 个已跟踪文件，没有未跟踪文件：

| 文件 | 主要变化 |
|---|---|
| `main/audio/audio_mp3_decode.c` | SD 卡 MP3 重采样路径增加软件增益和 int16 饱和裁剪 |
| `main/audio/audio_sr.c` | `LISTENING` 进入条件增加 `is_awake` 保护，WebSocket 音频发送只在 `LISTENING` 下执行 |
| `main/com/com_status.c` | 状态机层面拒绝未唤醒时进入 `LISTENING`，取消唤醒时回到 `IDLE` |
| `main/bsp/bsp_http.c` | HTTP 请求体创建、客户端创建、响应缓冲区等增加失败处理和释放 |
| `main/bsp/bsp_mutable_buffer.c` | 可变缓冲区改为 PSRAM 分配并增加空指针/扩容失败保护 |
| `main/bsp/bsp_wifi.c/.h` | 新增 `bsp_wifi_wait_connected()` 等待 WiFi 连接结果 |
| `main/bsp/bsp_ws.c/.h` | `bsp_ws_init()` 临时改为无参形式，回调赋值被注释 |
| `main/main.c` | 引入 HTTP/NVS/WebSocket/MP3 下载相关头文件，预留激活码与 WebSocket 启动流程 |

## 推荐方案：音频播放增益与唤醒状态隔离

当前改动的核心思路是把“是否允许进入监听状态”和“是否继续做语音检测”分开处理：

```text
AFE fetch
  -> VAD_SPEECH
  -> 未唤醒: 不进入 LISTENING，但允许返回 vad_speech 给 MultiNet 检测
  -> 已唤醒: 进入 LISTENING，并允许 WebSocket 音频上行
```

这样可以避免设备在未唤醒时因为普通人声直接进入 `LISTENING` 或向 WebSocket 发送音频，同时保留 MultiNet 对人声片段的识别能力。

SD 卡音乐播放侧，则只在 `/sdcard` 文件启用重采样时对重采样后的单声道 PCM 做软件增益：

```text
MP3 decode
  -> 判断 /sdcard 路径
  -> 如果采样率/声道不等于 16k mono，则重采样
  -> 立体声 downmix
  -> 应用 SDCARD_RESAMPLE_GAIN_Q8 增益
  -> clip_int16 饱和裁剪
  -> audio_write
```

## 详细设计

### SD 卡 MP3 软件增益

`main/audio/audio_mp3_decode.c` 新增：

| 符号 | 作用 |
|---|---|
| `SDCARD_RESAMPLE_GAIN_Q8` | 使用 Q8 定点表示增益，当前值为 `768` |
| `clip_int16()` | 将放大后的 `int32_t` 样本限制在 `INT16_MIN ~ INT16_MAX` |
| `apply_sdcard_resample_gain()` | 对重采样后的单声道样本应用增益并裁剪 |

当前宏值：

```c
#define SDCARD_RESAMPLE_GAIN_Q8 768
```

由于 Q8 基准是 `256`，因此当前实际增益为：

```text
768 / 256 = 3.0 倍
```

注意：如果目标是严格 2.5 倍，宏值应为 `640`。当前文档按实际代码记录为 3.0 倍。

增益应用位置位于重采样输出写入前：

```c
resample_buf[out_frames++] = apply_sdcard_resample_gain(mono_sample);
```

这意味着：

| 场景 | 是否受影响 |
|---|---|
| `/sdcard` 且需要重采样/转单声道 | 受影响，会被软件放大 |
| `/sdcard` 但已经是 16k mono | 不经过当前重采样增益路径 |
| `/spiffs` 提示音 | 不受该增益影响 |

### 语音状态机保护

`main/audio/audio_sr.c` 中，`switch_to_listening()` 增加未唤醒保护：

```c
if (!is_awake) {
    return;
}
```

`handle_vad_state()` 在检测到 `VAD_SPEECH` 后仍会更新 `last_voice_tick` 并调用 `switch_to_listening()`，但如果当前未唤醒，会直接返回 `vad_speech`：

```text
VAD_SPEECH + is_awake=false
  -> 不切 LISTENING
  -> 返回 true
  -> 后续 MultiNet 仍可检测
```

WebSocket 音频上行增加状态判断：

```c
if (com_status == LISTENING && g_audio_out_ringbuf != NULL && is_wsline == true) {
    xRingbufferSend(...);
}
```

因此未唤醒时即使检测到人声，也不会把 AFE 数据写入 WebSocket 输出 ringbuffer。

### 全局状态机兜底

`main/com/com_status.c` 增加两层保护：

| 位置 | 行为 |
|---|---|
| `com_set_awake(false)` | 如果当前处于 `LISTENING`，立即切回 `IDLE` |
| `com_status_change(LISTENING)` | 如果 `is_awake == false`，直接忽略状态切换 |

这相当于在状态机公共入口也加了一层约束，避免其他模块绕过 `audio_sr.c` 直接切入 `LISTENING`。

### HTTP 与可变缓冲区

`main/bsp/bsp_http.c` 主要改动：

| 改动 | 目的 |
|---|---|
| `protocol_http_get_body()` 检查 `cJSON_CreateObject()` 返回值 | 避免内存不足时空指针访问 |
| `http_request_active_code()` 检查 `bsp_mutable_buffer_create()` | 避免响应缓冲区创建失败后继续请求 |
| 检查 `esp_http_client_init()` 返回值 | 避免 HTTP client 创建失败后继续设置 header |
| 检查 body 创建结果 | 避免 `esp_http_client_set_post_field()` 使用空指针 |
| `esp_http_client_perform()` 返回错误时打印日志 | 便于定位网络/证书/服务端错误 |
| 请求完成后释放 `body`、cleanup client、释放 buffer | 减少内存泄漏 |

`main/bsp/bsp_mutable_buffer.c` 主要改动：

| 改动 | 目的 |
|---|---|
| 结构体和数据区使用 `heap_caps_malloc(..., MALLOC_CAP_SPIRAM)` | HTTP 响应缓存优先放到 PSRAM |
| `bsp_mutable_buffer_add_data()` 增加 `buffer/data/len` 检查 | 防止异常输入 |
| `heap_caps_realloc()` 失败时保持原 buffer | 避免扩容失败导致原数据丢失 |
| 每次追加后补 `'\0'` | 方便按字符串打印 HTTP 响应 |
| `bsp_mutable_buffer_free()` 支持空指针 | 调用侧释放更安全 |

### WiFi 等待接口与 WebSocket 入口

`main/bsp/bsp_wifi.c/.h` 新增：

```c
esp_err_t bsp_wifi_wait_connected(TickType_t ticks_to_wait);
```

该接口等待 `WIFI_CONNECTED_BIT` 或 `WIFI_FAIL_BIT`：

| 结果 | 返回值 |
|---|---|
| 已连接 | `ESP_OK` |
| 连接失败 | `ESP_FAIL` |
| 超时 | `ESP_ERR_TIMEOUT` |
| WiFi event group 未初始化 | `ESP_ERR_INVALID_STATE` |

`main/main.c` 中预留了以下流程，但目前是注释状态：

```text
等待 WiFi 连接，最长 HTTP_WIFI_WAIT_MS
  -> 请求激活码 http_request_active_code()
  -> 启动 WebSocket bsp_ws_start()
```

`main/bsp/bsp_ws.c/.h` 中 `bsp_ws_init()` 从带回调参数改为无参，原本的 `text_callback` 和 `bin_callback` 赋值被注释。这个改动会降低 WebSocket 初始化时注入回调的能力，需要确认后续是否由其他方式设置回调。

## 状态流转

### 唤醒状态与监听状态

```text
未唤醒 IDLE
  -> VAD 检测到人声
  -> 不进入 LISTENING
  -> MultiNet 可继续检测
  -> 不发送 WebSocket 音频

已唤醒
  -> VAD 检测到人声
  -> com_status_change(LISTENING)
  -> MultiNet 检测
  -> WebSocket ringbuffer 可发送音频

唤醒超时或 com_set_awake(false)
  -> 如果当前是 LISTENING
  -> 切回 IDLE
```

### SD 卡播放数据流

```text
audio_mp3_play_file_async("/sdcard/xxx.mp3")
  -> mp3_player_task()
  -> enable_resample = true
  -> mp3_decode_file()
  -> need_resample 判断采样率/声道
  -> downmix + resample
  -> apply_sdcard_resample_gain()
  -> audio_write()
```

## 风险评估

| 风险点 | 触发条件 | 影响 | 建议验证 |
|---|---|---|---|
| SD 卡增益过高导致削波 | 当前 `SDCARD_RESAMPLE_GAIN_Q8=768`，等于 3.0 倍 | 音乐响度提高，但大动态或本身音量高的曲目可能失真 | 用多首 SD 卡 MP3 测试是否有破音；如果目标是 2.5 倍，应改为 `640` |
| 16k mono 的 SD 卡 MP3 不被增益处理 | `need_resample == false` | 某些 SD 卡歌曲仍可能偏小 | 如需全路径增益，应在写入前统一处理 PCM |
| 未唤醒时 MultiNet 仍会检测 | VAD 检测到人声但 `is_awake=false` | CPU 仍会消耗在识别链路，但不会进入 `LISTENING` 或上行音频 | 串口观察未唤醒说话时状态是否保持 `IDLE` |
| `bsp_ws_init()` 回调被注释 | 后续 WebSocket 收到文本/二进制消息 | 可能无法执行业务回调 | 确认是否仍需要 `text_callback_t/bin_callback_t` |
| HTTP 响应缓冲区扩容失败静默返回 | PSRAM 不足或碎片化 | HTTP 响应可能不完整 | 在 buffer 扩容失败处增加错误日志 |
| main 中联网流程仍注释 | 需要自动请求激活码或启动 WebSocket | 实际运行不会执行 HTTP/WS 初始化流程 | 联调时显式启用注释代码并验证 WiFi 超时分支 |

## 与当前方案的映射

| 旧行为 | 当前行为 | 影响 |
|---|---|---|
| SD 卡重采样后直接写入 `resample_buf` | 写入前调用 `apply_sdcard_resample_gain()` | 提高 SD 卡音乐播放响度 |
| `VAD_SPEECH` 可直接触发 `LISTENING` | `LISTENING` 必须满足 `is_awake == true` | 未唤醒的人声不会改变对外状态 |
| WebSocket ringbuffer 只判断连接和 buffer | 增加 `com_status == LISTENING` 条件 | 未唤醒人声不会上行 |
| 状态切换入口不拦截未唤醒的 `LISTENING` | `com_status_change(LISTENING)` 会检查 `is_awake` | 防止其他模块误切状态 |
| HTTP 请求缺少部分失败释放路径 | 初始化/请求体/执行失败均有处理 | 降低空指针和泄漏风险 |
| WiFi 连接状态由调用方自行判断 | 新增 `bsp_wifi_wait_connected()` | 便于 main 或业务模块等待联网结果 |

## 迁移步骤

1. 烧录后先验证 SD 卡 MP3 播放音量，重点听是否有削波、破音或底噪抬升。
2. 如果目标增益是 2.5 倍，将 `SDCARD_RESAMPLE_GAIN_Q8` 从 `768` 调整为 `640`。
3. 未唤醒状态下说话，确认日志里不会出现 `status change: IDLE -> LISTENING`，同时不会向 WebSocket ringbuffer 写入音频。
4. 唤醒后说话，确认可以进入 `LISTENING`，并能继续执行 MultiNet 命令检测。
5. 如果要启用激活码请求和 WebSocket，恢复 `main.c` 中的 WiFi 等待、`http_request_active_code()` 和 `bsp_ws_start()` 调用。
6. 确认 `bsp_ws_init()` 是否仍需要文本/二进制回调参数；如果需要，恢复原签名或提供新的回调注册接口。
