# 语音命令播放与串口协议改动说明

## 背景

本次工作区相对 `HEAD` 共修改 10 个文件，主要围绕语音识别命令处理、MP3 播放中断、串口协议分流、SR 任务控制和启动流程做调整。

核心目标可以概括为两点：

1. 语音命令执行时，统一处理“发送床控串口命令 + 播放语音反馈 + 唤醒状态恢复”的流程。
2. 串口接收侧按帧头区分普通 `0xAA` 指令、WiFi 分包指令和 `0xBB` 控制指令，为后续状态查询或 OTA 类命令预留入口。

## 推荐方案：以播放状态作为语音交互的同步边界

当前改动引入了 `stop_play_mp3()`、`MP3_after_awake` 和 `play_mp3()` 三个关键机制：

| 机制 | 所在文件 | 作用 |
|---|---|---|
| `stop_play_mp3()` | `main/audio/audio_mp3_decode.c` | 请求当前 MP3 解码循环尽快退出，用于唤醒词打断正在播放的反馈音 |
| `MP3_after_awake` | `main/com/com_status.c` / `main/com/com_status.h` | 标记一次命令反馈播放结束后需要恢复唤醒态 |
| `play_mp3()` | `main/app/app_sr_handler.c` | 封装串口下发、反馈音播放和唤醒态临时切换 |

状态流转的意图是：

```text
IDLE -> LISTENING -> SPEAKING -> IDLE
                      |
                      +-- 命令反馈播放期间临时 com_set_awake(false)
                      +-- MP3 创建成功后通过 MP3_after_awake 恢复 com_set_awake(true)
```

唤醒词命中时新增 `stop_play_mp3()`，使新的唤醒反馈可以打断上一段 MP3，避免旧播报占用播放互斥锁导致新反馈被跳过。

## 架构总览

| 模块 | 改动重点 | 行为影响 |
|---|---|---|
| 语音命令处理 | 新增 `play_mp3()`，批量替换命令分支中的重复代码 | 大部分床控命令先临时清除唤醒态，再下发串口并异步播放反馈 |
| MP3 解码播放 | 新增停止请求标志，解码和写 PCM 前检查停止状态 | 支持在播放中途被唤醒词或其他逻辑打断 |
| 串口协议 | 使用帧头 `0xAA` / `0xBB` 分流，WiFi 分包只接受 `0xAA` 帧 | 避免 `0xBB` 帧误进入 WiFi 凭据拼包流程 |
| SR 检测 | 移除“播放中跳过识别”的分支，VAD 检测到语音即进入监听 | 播放期间也可能触发语音检测，配合 MP3 中断实现打断播放 |
| 启动流程 | 注释掉 WiFi 初始化和 OTA rollback 检查 | 当前启动路径更聚焦本地语音和串口，但联网/OTA 能力不会自动初始化 |

## 详细设计

### 1. 语音命令反馈封装

`main/app/app_sr_handler.c` 新增：

```c
static void play_mp3(const uint8_t *data, const char *file_name)
{
    com_set_awake(false);
    MP3_after_awake = true;
    usart_send_data(data);
    audio_mp3_play_file_async(file_name);
}
```

它替代了大量重复的：

```text
usart_send_data(...)
audio_mp3_play_file_async(...)
```

新的命令处理含义是：执行床控命令前暂时将 `is_awake` 置为 `false`，避免反馈音播放过程中继续保持唤醒窗口；随后通过 `MP3_after_awake` 在 MP3 播放任务创建成功后恢复唤醒态。

唤醒词命令 `0/1/2` 在未唤醒和已唤醒分支中都增加了 `stop_play_mp3()`，用于打断正在播放的 MP3 并播放唤醒反馈 `105.mp3` 或 `103.mp3`。

### 2. MP3 播放可中断

`main/audio/audio_mp3_decode.c` 新增 `s_mp3_stop_requested`：

| 位置 | 改动 |
|---|---|
| `stop_play_mp3()` | 播放中将 `s_mp3_stop_requested` 置为 `true` |
| `mp3_decode_file()` | 外层读文件循环、内层解码循环、PCM 写出前都检查停止请求 |
| `mp3_player_task()` | 停止请求视为正常结束，并切回 `IDLE` |
| `audio_mp3_play_file_async()` | 新播放开始前清除停止标志 |

这让播放任务不再只能依赖文件播完或解码错误结束，而是可以由业务侧主动停止。

同时，原先 `audio_mp3_play_file_async()` 中“`com_status == LISTENING` 时跳过 MP3”的保护被移除。结合 SR 侧移除“播放中跳过识别”，当前设计更偏向支持播放期间继续监听并允许唤醒词打断。

### 3. 串口协议分流

`main/app/app_usart_protocol.c` 新增 `WIFI_FRAME_HEADER 0xAA` 和 `is_wifi_fragment_packet()`，将 WiFi 分包判断从“只看命令字”收紧为：

```text
packet[0] == 0xAA && cmd_id in {0x00, 0x02}
```

普通命令处理新增 `header` 参数：

| 帧头 | 当前处理 |
|---|---|
| `0xAA` | 进入语音提示播放分支，命令 `0x6B/0x6D/0x6F/0x71/0x73/0x75/0x77/0x80/0x81` 对应 `200.mp3` 到 `208.mp3` |
| `0xBB` | 预留 `0x3E` 子状态处理；`0x99` 触发 `start_xiaozhi_ota()` |

需要注意：旧代码注释表示曾按 `0x6A ~ 0x76` 的偶数命令用数学规律映射 MP3；当前实现改为显式处理奇数命令 `0x6B ~ 0x77` 以及 `0x80/0x81`。这属于协议行为变更，需要和上位机或主控协议确认。

### 4. SR 检测任务控制

`main/audio/audio_sr.c` 新增三个任务句柄：

```text
audio_feed_task_handle
audio_detect_task_handle
sr_handler_task_handle
```

并导出：

```text
app_sr_suspend_tasks()
app_sr_resume_tasks()
```

设计意图是允许外部模块暂停/恢复 SR 的 feed、detect、handler 三个任务。但当前 `sr_handler_task` 创建时仍传入 `NULL` 作为任务句柄输出，导致 `sr_handler_task_handle` 实际不会被赋值。

### 5. 启动流程调整

`main/main.c` 当前注释掉：

```text
bsp_wifi_init()
check_ota_rollback()
```

保留串口初始化、音频初始化、SR 启动和启动提示音播放。

这会让设备启动后不主动初始化 WiFi，也不会执行 OTA rollback 检查；如果当前目标是离线语音床控调试，这是合理的临时入口。如果仍要求 OTA 或 WiFi 配网可用，需要恢复或迁移初始化时机。

## 可删除或可收敛的旧代码

| 可收敛项 | 所在文件 | 规模 | 原因 |
|---|---|---:|---|
| 重复的 `usart_send_data()` + `audio_mp3_play_file_async()` 分支 | `main/app/app_sr_handler.c` | `~100 行` | 已由 `play_mp3()` 承接统一流程 |
| 播放中强制跳过 SR 检测的逻辑 | `main/audio/audio_sr.c` | `~10 行` | 当前设计允许播放期间继续监听并打断 MP3 |
| `com_status == LISTENING` 时跳过 MP3 的保护 | `main/audio/audio_mp3_decode.c` | `~5 行` | 当前播放策略改为可中断，而不是直接拒绝播放 |
| 数学规律映射 `0x6A ~ 0x76` MP3 的旧逻辑 | `main/app/app_usart_protocol.c` | `~8 行` | 已替换为显式命令映射，但需要确认协议新旧差异 |

## 风险评估

| 风险点 | 触发条件 | 影响 | 建议验证 |
|---|---|---|---|
| `sr_handler_task_handle` 未赋值 | 调用 `app_sr_suspend_tasks()` / `app_sr_resume_tasks()` | 对空句柄执行 suspend/resume，可能触发异常或无效操作 | 将 `xTaskCreatePinnedToCore(..., &sr_handler_task_handle, ...)` 后再测试 |
| `MP3_after_awake` 是全局布尔值 | 多处同时触发播放或播放创建失败 | 唤醒态恢复可能与实际播放状态不一致 | 验证播放 busy、文件不存在、任务创建失败场景 |
| 播放中仍启用 SR 检测 | 反馈音被麦克风采集 | 可能出现反馈音误触发 VAD 或命令识别 | 实机验证外放音量、AEC/AFE 配置和误唤醒率 |
| 串口 MP3 命令映射变化 | 上位机仍发送旧的偶数 `0x6A/0x6C/...` | 原有提示音不再播放 | 用协议表逐条回归 `0x6A ~ 0x81` |
| WiFi/OTA 初始化被注释 | 需要配网、OTA 或 rollback 检查 | 相关能力启动后不可用或延后可用 | 明确当前固件是否为离线调试版本 |
| `0xBB/0x3E` 分支为空 | 主控发送状态查询或控制子命令 | 命令被消费但无业务效果 | 补齐 `packet[3] == 0x01/0x02` 的具体处理 |

## 与当前方案的映射

| 原实现 | 新实现 | 说明 |
|---|---|---|
| 床控命令各自调用串口发送和 MP3 播放 | `play_mp3(data, file_name)` | 将唤醒态、串口发送、播放启动合并到统一入口 |
| 播放期间 SR 直接退回或跳过 | SR 持续处理 VAD/MultiNet | 为语音打断播放提供基础 |
| MP3 播放不可主动停止 | `stop_play_mp3()` + `s_mp3_stop_requested` | 解码循环可在下一次检查点退出 |
| WiFi 分包只按 `cmd_id` 判断 | WiFi 分包按 `0xAA` 帧头和 `cmd_id` 判断 | 避免其他帧头的相同命令字被误判 |
| 串口音频提示用范围公式映射 | 显式 `switch` 映射 | 更直观，但需要维护协议表 |

## 迁移步骤

1. 确认串口协议表：重点核对 `0xAA` 下 `0x6B/0x6D/.../0x81` 与 MP3 文件 `200.mp3` 到 `208.mp3` 的对应关系。
2. 修正 SR 任务句柄：将 `sr_handler_task` 创建的句柄参数改为 `&sr_handler_task_handle`，并在 suspend/resume 前判断句柄非空。
3. 明确启动模式：如果当前固件仍要求联网或 OTA，恢复 `bsp_wifi_init()` 和 `check_ota_rollback()`，或移动到明确的条件分支。
4. 补齐 `0xBB/0x3E` 分支：定义 `packet[3] == 0x01` 和 `0x02` 的业务含义，避免空处理。
5. 做实机回归：覆盖唤醒词打断播放、播放 busy、MP3 文件缺失、语音命令连续触发、WiFi 分包配网、`0x99` OTA 触发。

## 验证方式

建议最小验证集：

1. 播放 `107.mp3` 启动提示期间说唤醒词，确认旧播放可停止，新唤醒提示能播放。
2. 唤醒后执行床头/床尾/灯光/按摩/温度等至少 5 类命令，确认串口数据发送和对应 MP3 反馈都正常。
3. 连续快速触发两个命令，确认播放互斥锁不会导致状态长期卡在 `SPEAKING`。
4. 发送 WiFi SSID/Password 分包，确认只有 `0xAA` 帧进入拼包并最终调用 `bsp_wifi_start_connect()`。
5. 发送 `0xBB 0x99` OTA 指令，确认 `start_xiaozhi_ota()` 被触发。
6. 调用 `app_sr_suspend_tasks()` / `app_sr_resume_tasks()`，确认三个任务句柄均有效。

## 当前结论

本次改动的主线是把语音反馈从“播完才能继续”的单向流程，调整为“播放可被语音打断，并用唤醒态标志协调播放前后状态”的交互流程。整体方向更适合语音控制设备的连续交互，但还需要补齐 SR handler 任务句柄、串口协议表确认和 `0xBB/0x3E` 空分支，才能作为稳定版本合入。
