# SD 卡 MP3 下载刷新与 OTA JSON 地址改造说明

## 背景

本次工作区改动围绕三个方向展开：

1. 串口 `0xBB` 控制命令中的 MP3 下载/播放链路从 SPIFFS 示例文件切换到 SD 卡文件。
2. `app_mp3_download_start_async()` 下载成功后，如果目标文件位于 `/sdcard/`，自动刷新 `sd_file_names` 与 `sd_file_count`。
3. OTA 升级从固定固件 URL 改为读取 JSON 中的 `url` 字段，便于服务端动态下发固件地址。

当前相对 `HEAD` 涉及 6 个文件：

| 文件 | 变更规模 | 主要内容 |
|---|---:|---|
| `main/app/app_MP3_download.c` | `~15 行` | SD 卡下载完成后刷新 MP3 文件列表 |
| `main/app/app_usart_protocol.c` | `~15 行` | `0x91/0x92` 命令改为下载/播放 `/sdcard/yueguang.mp3` |
| `main/ota.c` | `~53 行` | OTA JSON 缓冲区扩大、循环读取、解析 `url` 字段 |
| `main/ota.h` | `~6 行` | 新增 OTA JSON 缓冲区和 URL 长度宏，移除固定固件 URL |
| `main/bsp/bsp_usart.h` | `~6 行` | UART 从 `UART_NUM_0/GPIO43/44` 切换到 `UART_NUM_1/GPIO17/18` |
| `main/audio/audio_offline_sr_handler.c` | `~6 行` | 唤醒/进入工作态提示音从 `103.mp3` 改为 `104.mp3` |

## 推荐方案：下载完成后刷新 SD 卡 MP3 索引

原有 SD 卡播放逻辑依赖全局数组：

```c
sd_file_names[MAX_FILES_TO_LIST][MAX_FILENAME_LEN]
sd_file_count
```

这些数组只在启动时通过 `sd_list_files("/sdcard")` 初始化。下载任务在运行期新增 MP3 文件后，如果不重新扫描目录，语音控制里的上一首/下一首/播放当前 SD 歌曲仍然只能看到旧列表。

本次改动把刷新动作放在 `mp3_download_task()` 下载成功分支中：

```text
app_mp3_download_start_async()
  -> mp3_download_task()
     -> download_audio_to_vfs()
     -> 成功且 file_path 以 /sdcard/ 开头
     -> sd_list_files("/sdcard")
     -> 更新 sd_file_names / sd_file_count
```

## 为什么选这个方案

| 方案 | 优点 | 取舍 |
|---|---|---|
| 下载成功后调用 `sd_list_files("/sdcard")` | 与启动扫描逻辑复用，行为一致 | 每次下载完成会重新遍历 SD 根目录 |
| 手动追加新文件名到 `sd_file_names` | 扫描开销更小 | 需要额外处理重复文件、删除失败、大小写后缀等边界 |
| 播放前临时扫描目录 | 列表最新 | 播放命令路径更重，用户交互时延更不可控 |

当前改动选择第一种方案，适合现阶段功能验证：实现简单，和已有列表生成规则保持一致。

## 架构总览

### MP3 下载链路

```text
USART BB 04 91 <xor>
  -> app_usart_protocol_handle_packet()
  -> process_normal_command()
  -> app_mp3_download_start_async(".../yueguang.mp3", "/sdcard/yueguang.mp3")
  -> mp3_download_task()
  -> download_audio_to_vfs()
  -> sd_list_files("/sdcard")
```

### MP3 播放链路

```text
USART BB 04 92 <xor>
  -> app_usart_protocol_handle_packet()
  -> process_normal_command()
  -> audio_mp3_play_file_async("/sdcard/yueguang.mp3")
```

### OTA 链路

```text
start_xiaozhi_ota()
  -> xiaozhi_ota_task()
  -> GET OTA_VERSION_URL
  -> 读取 JSON 到 OTA_JSON_BUFFER_SIZE 缓冲区
  -> cJSON_Parse()
  -> 读取 version 与 url
  -> 版本比较
  -> esp_https_ota(config.url = JSON url)
```

## 详细设计

### 1. SD 卡 MP3 下载后刷新

`main/app/app_MP3_download.c` 新增：

```c
static bool is_sdcard_file_path(const char *file_path)
{
    return file_path != NULL && strncmp(file_path, "/sdcard/", strlen("/sdcard/")) == 0;
}
```

下载任务成功后增加路径判断：

```c
if (is_sdcard_file_path(request->file_path)) {
    sd_list_files("/sdcard");
}
```

这个判断使用 `/sdcard/` 而不是 `/sdcard`，可以避免误匹配 `/sdcard_backup/...` 一类路径。

### 2. 串口命令 0x91/0x92 行为调整

`main/app/app_usart_protocol.c` 中 `0xBB` 普通命令分支调整：

| 命令 | 原行为 | 新行为 |
|---|---|---|
| `0x91` | 下载 `welcome.mp3` 到 `/spiffs/welcome.mp3` | 下载 `yueguang.mp3` 到 `/sdcard/yueguang.mp3` |
| `0x92` | 播放 `/spiffs/welcome.mp3` | 播放 `/sdcard/yueguang.mp3` |

对应校验帧仍按 XOR 规则发送：

```text
BB 04 91 2E
BB 04 92 2D
```

### 3. OTA JSON 动态 URL

`main/ota.h` 移除固定固件 URL：

```c
#define OTA_URL "http://10.176.68.23:8000/xiaozhi_update.bin"
```

新增：

```c
#define OTA_VERSION_URL "http://192.168.26.2:8080/xiaozhi_update.json"
#define OTA_JSON_BUFFER_SIZE 512
#define OTA_URL_MAX_LEN 256
```

`main/ota.c` 主要变化：

| 变更点 | 说明 |
|---|---|
| 检查 `esp_http_client_init()` 返回值 | 避免 HTTP client 创建失败后继续使用空句柄 |
| JSON 缓冲区从 `128` 扩到 `OTA_JSON_BUFFER_SIZE` | 容纳 `version` 与 `url` 等字段 |
| 循环读取 JSON 响应 | 避免一次 `esp_http_client_read()` 没读完整 |
| 新增 `url` 字段解析 | OTA 固件地址由服务端 JSON 下发 |
| `config.url = ota_url` | 实际升级 URL 不再使用编译期固定宏 |

预期 JSON 结构类似：

```json
{
  "version": "1.0.1",
  "url": "http://192.168.26.2:8080/xiaozhi_update.bin"
}
```

### 4. UART 引脚迁移

`main/bsp/bsp_usart.h` 将串口从默认 `UART_NUM_0` 切换为 `UART_NUM_1`：

| 配置项 | 原值 | 新值 |
|---|---|---|
| `UART_NUM` | `UART_NUM_0` | `UART_NUM_1` |
| `UART_TX_IO` | `GPIO_NUM_43` | `GPIO_NUM_17` |
| `UART_RX_IO` | `GPIO_NUM_44` | `GPIO_NUM_18` |

这会改变外部串口接线要求，需要同步硬件连接和上位机串口配置。

### 5. 离线语音提示音调整

`main/audio/audio_offline_sr_handler.c` 中 `case 0/1/2` 的提示音从：

```text
/spiffs/103.mp3
```

调整为：

```text
/spiffs/104.mp3
```

该改动影响语音识别进入唤醒/工作相关分支时的播报内容。

## 可删除或后续可收敛的代码

| 可收敛项 | 所在文件 | 规模 | 原因 |
|---|---|---:|---|
| 编译期固定 OTA 固件 URL | `main/ota.h` | `1 行` | 已由 JSON 中的 `url` 字段替代 |
| `0x91/0x92` 对 `/spiffs/welcome.mp3` 的测试路径 | `main/app/app_usart_protocol.c` | `~2 行` | 当前命令已切换为 SD 卡文件测试 |
| `app_usart_protocol.c` 中乱码中文注释/日志 | `main/app/app_usart_protocol.c` | `~4 行` | 不影响当前构建，但影响维护和审阅 |

## 风险评估

| 风险点 | 触发条件 | 影响 | 建议验证 |
|---|---|---|---|
| 下载成功但列表刷新失败 | SD 卡未挂载、目录不可打开、内存不足 | `sd_file_names` 不包含新下载文件 | 观察 `failed to refresh sd_file_names after download` 日志 |
| SD 列表刷新与播放并发 | 下载任务刷新列表时，语音任务正在读取 `sd_file_names` | 可能读到刷新中的列表状态 | 如频繁下载/播放，建议后续给 SD 列表加互斥锁 |
| URL 中包含中文 | HTTP URL 使用 `/月光.mp3` 等非 ASCII 路径 | `esp_http_client` 解析 URL 失败 | URL 使用英文名或百分号编码 |
| UART 引脚切换 | 硬件仍接在 GPIO43/44 | 串口命令无法收到 | 确认上位机接线改到 TX17/RX18 对应关系 |
| OTA JSON 过大 | JSON 超过 `512` 字节 | JSON 截断导致解析失败 | 保持 JSON 精简，或继续增大 `OTA_JSON_BUFFER_SIZE` |
| `app_usart_protocol.c` 注释/日志乱码 | 文件编码混用或编辑器编码不一致 | 审阅困难，后续编辑可能引入乱码扩散 | 建议统一保存为 UTF-8，无 BOM，并清理乱码注释 |

## 当前方案映射

| 当前实现 | 新方案 | 说明 |
|---|---|---|
| 启动时扫描 SD 卡 MP3 | 启动扫描 + 下载成功后刷新 | 运行期新增文件可以进入播放列表 |
| `/spiffs/welcome.mp3` 下载测试 | `/sdcard/yueguang.mp3` 下载测试 | 验证 SD 卡写入和播放链路 |
| 固定 `OTA_URL` | JSON 下发 `url` | 服务端可动态控制固件下载地址 |
| `UART_NUM_0` GPIO43/44 | `UART_NUM_1` GPIO17/18 | 避免占用默认串口或适配新硬件接线 |

## 迁移步骤

1. 确认串口硬件连接已切换到 `UART_NUM_1` 的 GPIO17/GPIO18。
2. 服务端准备 `http://192.168.26.2:8080/yueguang.mp3`，文件名建议保持 ASCII。
3. 发送 `BB 04 91 2E` 触发下载，观察下载成功和 `sd_list_files("/sdcard")` 刷新日志。
4. 发送 `BB 04 92 2D` 播放 `/sdcard/yueguang.mp3`。
5. 服务端准备 `xiaozhi_update.json`，至少包含 `version` 和 `url` 字段。
6. 发送 OTA 触发命令，确认日志中打印 `OTA URL: ...`，并进入固件下载。
