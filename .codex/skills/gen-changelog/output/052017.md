# 当前代码改动说明

## 背景

本次改动围绕 ESP32-S3 端启动流程、串口配网、WiFi 自动连接和调试日志进行整理。核心目标是降低开机阻塞时间，保证串口下发的新 WiFi 只有在真实连接成功后才保存到 NVS，同时把 `bsp_wifi.c` 和 `usart_init.c` 中较集中的逻辑拆成更清晰的小函数。

当前工作区相对 `HEAD` 涉及 4 个文件：

| 文件 | 变更规模 | 主要内容 |
|---|---:|---|
| `main/bsp/bsp_wifi.c` | `~283 行` | WiFi 初始化、历史自动重连、串口配网和 NVS 保存逻辑重构 |
| `main/usart/usart_init.c` | `~268 行` | 串口收包、校验、WiFi 分片拼接逻辑拆分 |
| `main/main.c` | `~23 行` | 启动提示音 `107.mp3` 改为命令词加载完成后播放 |
| `main/com/com_debug.h` | `~2 行` | `DEBUG` 从 `1` 改为 `0`，关闭 `MY_LOGx` 宏日志 |

## 推荐方案：异步启动与确认后持久化

新方案把启动阶段拆成三类互不阻塞的动作：

1. `bsp_wifi_init()` 只初始化 WiFi 驱动，并对历史 WiFi 发起后台自动连接，不再等待连接成功或失败。
2. 串口配网收到完整 SSID/密码后，先使用 `WIFI_STORAGE_RAM` 临时连接。
3. 只有收到 `IP_EVENT_STA_GOT_IP` 后，才把当前成功连接的 WiFi 配置切换到 `WIFI_STORAGE_FLASH` 并写入 NVS。

这样可以避免旧 WiFi 连接失败时拖慢开机，也避免错误的新 WiFi 覆盖 NVS 中可用的历史配置。

## 架构总览

```text
app_main()
  -> bsp_wifi_init()
       -> 初始化 WiFi STA
       -> 如果 NVS 有历史 SSID，只发起后台 esp_wifi_connect()
  -> usart_init()
       -> 创建串口事件任务
  -> audio_init()
  -> app_sr_start()
       -> 加载模型
       -> load_speech_commands()
       -> 创建语音识别任务
  -> play_startup_prompt()
       -> 播放 107.mp3
```

串口配网数据流：

```text
UART_DATA
  -> usart_receive_callback()
  -> usart_read_packet()
  -> usart_packet_checksum_passed()
  -> usart_dispatch_packet()
  -> process_wifi_fragments()
  -> bsp_wifi_start_connect()
```

WiFi 状态流转：

```text
历史配置: INIT -> STA_START -> 后台 CONNECTING -> GOT_IP 或 DISCONNECTED 重试
串口配网: UART 完整分片 -> RAM 配置 -> CONNECTING -> GOT_IP -> 保存 NVS
```

## 详细设计

### WiFi 初始化与历史连接

`main/bsp/bsp_wifi.c` 将原本集中在 `wifi_init_sta()` 和事件回调里的逻辑拆成以下职责：

| 函数 | 职责 |
|---|---|
| `wifi_init_driver()` | 初始化 `esp_netif`、事件循环、默认 STA、WiFi driver 和 STA 模式 |
| `wifi_register_event_handlers()` | 注册 `WIFI_EVENT` 和 `IP_EVENT_STA_GOT_IP` 回调 |
| `wifi_auto_connect_saved_config()` | 检查 NVS 历史 SSID，有则后台发起 `esp_wifi_connect()` |
| `wifi_retry_or_fail()` | 处理断开后的重试和失败事件位 |
| `wifi_save_current_config_to_nvs()` | 连接成功后保存当前 WiFi 配置到 NVS |

关键行为变化是移除了历史连接中的阻塞等待：

```text
旧实现: esp_wifi_connect() -> xEventGroupWaitBits(..., portMAX_DELAY)
新实现: esp_wifi_connect() -> 立即返回，系统继续启动
```

这解决了旧路由器不可用或密码变更时，开机被 WiFi 重试流程拖慢的问题。

### 串口配网与 NVS 保存

`bsp_wifi_start_connect()` 现在使用串口下发的 SSID/密码进行临时连接：

```text
esp_wifi_disconnect()
esp_wifi_stop()
esp_wifi_set_storage(WIFI_STORAGE_RAM)
esp_wifi_set_config(WIFI_IF_STA, &wifi_config)
esp_wifi_start()
esp_wifi_connect()
```

只有连接成功并拿到 IP 后，`wifi_handle_got_ip()` 才调用 `wifi_save_current_config_to_nvs()`。因此：

```text
新 WiFi 可用 -> 保存到 NVS
新 WiFi 不可用 -> 不覆盖 NVS
```

为了避免串口配网切换配置时被旧连接的断开事件抢回重连，代码保留 `s_uart_provisioning` 标志。在串口配网更新配置期间，`WIFI_EVENT_STA_DISCONNECTED` 会跳过自动重连。

### 串口收包与 WiFi 分片

`main/usart/usart_init.c` 保留原有协议，同时把处理步骤拆开：

| 函数 | 职责 |
|---|---|
| `usart_read_packet()` | 从 UART 读取以 `0xAA` 开头的一帧数据 |
| `usart_packet_checksum_passed()` | 校验帧尾 XOR |
| `usart_dispatch_packet()` | 根据 `cmd_id` 分发命令 |
| `process_wifi_fragments()` | 处理 WiFi SSID/密码分片 |
| `wifi_fragment_connect_if_complete()` | 拼齐后触发 `bsp_wifi_start_connect()` |

协议保持不变：

| 字段 | 含义 |
|---|---|
| `0xAA` | 帧头 |
| `packet_len` | 整帧长度 |
| `cmd_id=0x00` | SSID |
| `cmd_id=0x02` | 密码 |
| `seq=0x01` | 第一段 15 字节 |
| `seq=0x02` | 第二段 15 字节 |
| 最后一字节 | XOR 校验 |

分片完成条件仍然是 `packet_mask == 0x0F`。如果第一段中遇到 `0x00`，仍会提前标记第二段完成，支持短 SSID/密码两包触发配网。

### 启动提示音播放时机

`main/main.c` 中 `107.mp3` 的播放从 `app_sr_start()` 之前移动到之后：

```text
旧流程: audio_init() -> 播放 107.mp3 -> 延时 7500ms -> app_sr_start()
新流程: audio_init() -> app_sr_start() -> 播放 107.mp3
```

由于 `app_sr_start()` 内部会先执行 `load_speech_commands()`，新流程保证启动提示音在命令词加载完成后播放。同时删除固定 `7500ms` 延时，减少启动阻塞。

### 日志开关

`main/com/com_debug.h` 中：

```c
#define DEBUG 0
```

这会关闭 `MY_LOGE/MY_LOGW/MY_LOGI/MY_LOGD/MY_LOGV` 宏展开。需要注意，直接调用 `ESP_LOGx(TAG, ...)` 的日志不受这个宏控制。

## 可删除的代码

| 可删除 | 所在文件 | 行数 | 原因 |
|---|---|---:|---|
| 历史 WiFi 自动重连中的 `xEventGroupWaitBits(..., portMAX_DELAY)` 阻塞等待 | `main/bsp/bsp_wifi.c` | `~15 行` | 后台自动连接后不再需要阻塞启动流程 |
| `STARTUP_MP3_WAIT_MS` 和启动阶段固定延时 | `main/main.c` | `~8 行` | 提示音改为异步播放，不再用固定等待控制启动顺序 |
| 串口收包函数中的内联校验和分发大块逻辑 | `main/usart/usart_init.c` | `~60 行` | 已拆分为 `usart_read_packet()`、`usart_packet_checksum_passed()`、`usart_dispatch_packet()` |
| WiFi 分片中的局部静态散变量 | `main/usart/usart_init.c` | `~5 行` | 已收敛到 `wifi_fragment_context_t s_wifi_fragments` |

## 吞吐量/资源/风险评估

| 风险点 | 触发条件 | 影响 | 建议验证 |
|---|---|---|---|
| 历史 WiFi 后台连接与串口配网同时发生 | 开机后旧 WiFi 正在重连时收到串口新 WiFi | 可能触发断开事件和配置切换竞争 | 开机后立即下发串口配网，确认无 `sta is connecting, cannot set config` |
| `DEBUG=0` 关闭宏日志 | 依赖 `MY_LOGI` 排查运行问题 | 日志减少，问题定位成本上升 | 联调阶段临时改回 `DEBUG=1` |
| 新 WiFi 失败不写 NVS | 串口下发错误 SSID/密码 | 重启后仍使用旧 NVS 配置 | 下发错误密码后重启，确认仍读取旧 SSID |
| 提示音播放与识别任务并行 | `app_sr_start()` 后立即播放 `107.mp3` | 识别任务启动后会遇到播放中状态 | 确认 `audio_mp3_is_playing()` 期间检测任务不会误识别 |

## 与当前方案的映射

| 当前实现 | 新方案 | 说明 |
|---|---|---|
| WiFi 初始化等待历史连接结果 | WiFi 初始化只发起后台连接 | 开机不再被旧 WiFi 失败阻塞 |
| 串口新 WiFi 可能提前写入 flash | 串口新 WiFi 先写 RAM，成功后写 flash | 保护 NVS 中已有可用配置 |
| WiFi 事件处理逻辑集中在 `event_handler()` | 事件回调转发到小函数 | 降低维护成本 |
| 串口收包、校验、分发混在一个函数 | 收包、校验、分发、拼片拆分 | 协议不变，代码路径更清晰 |
| 开机先播放 `107.mp3` 再加载命令词 | 命令词加载完成后播放 `107.mp3` | 启动提示音表达“系统已准备好”的语义更准确 |

## 迁移步骤

1. 保持串口协议不变，继续按 `cmd_id=0x00/0x02` 和 `seq=0x01/0x02` 下发 SSID/密码。
2. 联调时打开 `DEBUG=1` 观察配网日志，确认后再改回 `DEBUG=0`。
3. 用正确 WiFi 信息配网，确认日志出现 `got ip` 和 `SSID: xxx 已保存到 NVS`。
4. 用错误 WiFi 信息配网，确认连接失败后重启仍读取旧 NVS 配置。
5. 重启设备，确认 `107.mp3` 在命令词加载完成后播放，且不再固定等待 7.5 秒。
