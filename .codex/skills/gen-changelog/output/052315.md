# 串口触发 MP3 下载与播放改造说明

## 背景

本次改动的直接背景是串口命令 `0x91` 触发 HTTP 下载 MP3 时，系统出现：

```text
***ERROR*** A stack overflow in task usart_event has been detected.
```

原因是原实现把 `download_audio_to_vfs()` 直接放在 `usart_event` 串口事件任务中同步执行。该任务栈只有 `4 * 1024`，但下载路径会叠加串口接收缓冲、日志缓冲、HTTP client 调用链和网络临时缓冲，容易超过任务栈上限。

本次改造目标是：串口任务只负责识别命令并分发，HTTP 下载和文件写入移动到独立 FreeRTOS 任务中执行。

## 推荐方案：串口轻量分发 + 独立下载任务

新的处理方式：

```text
usart_event_task
  -> 收包/校验
  -> app_usart_protocol_handle_packet()
  -> 识别 BB 04 91 2E
  -> app_mp3_download_start_async()
  -> 创建 mp3_download 任务
  -> 串口任务立即返回

mp3_download
  -> mount_storage_partition()
  -> download_audio_to_vfs()
  -> HTTP GET
  -> 写入 /spiffs/welcome.mp3
  -> 释放资源
  -> vTaskDelete(NULL)
```

播放路径保持异步：

```text
BB 04 92 2D
  -> audio_mp3_play_file_async("welcome.mp3")
```

## 指令定义

| 功能 | 串口帧 | 说明 |
|---|---|---|
| 下载 MP3 | `BB 04 91 2E` | 下载 `http://192.168.26.3:8080/welcome.mp3` 到 `/spiffs/welcome.mp3` |
| 播放 MP3 | `BB 04 92 2D` | 播放 `/spiffs/welcome.mp3` |

校验规则仍为原串口驱动中的异或校验：

```text
checksum = header ^ length ^ command
```

因此：

```text
BB ^ 04 ^ 91 = 2E
BB ^ 04 ^ 92 = 2D
```

## 详细设计

### 1. 新增 MP3 下载模块

文件：[main/app/app_MP3_download.c](D:/working/xiaozhi1.0.1/main/app/app_MP3_download.c:1)

新增核心能力：

- `app_mp3_download_start_async()`：对外异步启动下载任务。
- `mp3_download_task()`：独立 FreeRTOS 下载任务。
- `download_audio_to_vfs()`：执行 HTTP 下载并写入 VFS 文件。

下载任务参数使用结构体保存：

```c
typedef struct
{
    char url[MP3_DOWNLOAD_URL_MAX_LEN];
    char file_path[MP3_DOWNLOAD_PATH_MAX_LEN];
} mp3_download_request_t;
```

这样串口命令中的 URL 和目标路径会被复制到任务参数里，避免传入临时指针导致生命周期问题。

### 2. 下载任务栈与并发保护

文件：[main/app/app_MP3_download.h](D:/working/xiaozhi1.0.1/main/app/app_MP3_download.h:11)

新增下载任务配置：

```c
#define MP3_DOWNLOAD_TASK_STACK_SIZE (8 * 1024)
#define MP3_DOWNLOAD_TASK_PRIORITY   4
```

同时新增下载状态位：

```c
static volatile bool s_mp3_downloading = false;
```

状态流转：

```text
空闲
  -> 收到 0x91
  -> s_mp3_downloading = true
  -> 创建 mp3_download
  -> 下载完成/失败
  -> s_mp3_downloading = false
  -> 空闲
```

如果下载中再次收到 `0x91`，函数返回 `ESP_ERR_INVALID_STATE`，避免多个任务同时写同一个 `/spiffs/welcome.mp3`。

### 3. SPIFFS 挂载前置

文件：[main/app/app_MP3_download.c](D:/working/xiaozhi1.0.1/main/app/app_MP3_download.c:18)

下载任务开始后先调用：

```c
mount_storage_partition();
```

这样可以避免直接 `fopen("/spiffs/welcome.mp3", "wb")` 时因为 `/spiffs` 未挂载而失败。

### 4. 网络缓冲从栈迁移到 heap

原下载函数中存在局部数组：

```c
char tmp_network_buf[NETWORK_CHUNK_SIZE];
```

该数组大小为 `2048` 字节，放在任务栈上会增加栈压力。新实现改为：

```c
network_buf = heap_caps_malloc(NETWORK_CHUNK_SIZE, MALLOC_CAP_DEFAULT);
```

PSRAM 大缓冲仍然使用：

```c
heap_caps_malloc(PSRAM_BUFFER_SIZE, MALLOC_CAP_SPIRAM);
```

资源分配策略：

| 缓冲 | 大小 | 分配区域 | 用途 |
|---|---:|---|---|
| `psram_buf` | `64KB` | PSRAM | 批量缓存网络数据，减少频繁写 Flash |
| `network_buf` | `2KB` | heap | 单次 HTTP read buffer |

### 5. 二进制写入模式

文件：[main/app/app_MP3_download.c](D:/working/xiaozhi1.0.1/main/app/app_MP3_download.c:105)

文件打开方式从文本写入改为二进制写入：

```c
fopen(file_path, "wb");
```

MP3 是二进制文件，使用 `"wb"` 更符合语义，也避免不同 C 库实现下文本模式对换行等内容做不必要处理。

### 6. 串口协议接入

文件：[main/app/app_usart_protocol.c](D:/working/xiaozhi1.0.1/main/app/app_usart_protocol.c:165)

新增 `BB` 帧命令：

```c
case 0X91:
    MY_LOGI("开始下载音频文件");
    app_mp3_download_start_async("http://192.168.26.3:8080/welcome.mp3", "/spiffs/welcome.mp3");
    break;
case 0X92:
    audio_mp3_play_file_async("welcome.mp3");
    break;
```

`0x91` 现在只负责启动下载任务，不再直接执行下载。`0x92` 继续使用现有 MP3 异步播放接口。

## 当前方案映射

| 旧实现 | 新实现 | 影响 |
|---|---|---|
| `0x91` 在 `usart_event` 内直接调用 `download_audio_to_vfs()` | `0x91` 调用 `app_mp3_download_start_async()` 创建独立任务 | 避免串口任务栈溢出 |
| 下载函数栈上分配 `2KB` 网络缓冲 | 网络缓冲改为 heap 分配 | 降低下载任务栈压力 |
| 调用方需要确保 `/spiffs` 已挂载 | 下载任务内部调用 `mount_storage_partition()` | 降低调用约束 |
| 可能重复触发多个下载 | `s_mp3_downloading` 拦截并发下载 | 避免同一文件被并发写入 |
| 文本方式打开 MP3 文件 | `fopen(..., "wb")` | 更适合二进制音频文件 |

## 资源与风险评估

| 风险点 | 触发条件 | 影响 | 建议验证 |
|---|---|---|---|
| 目标 URL 硬编码 | 服务器 IP 或端口变化 | 下载失败 | 烧录前确认 PC/服务器 IP 为 `192.168.26.3:8080` |
| `/spiffs` 剩余空间不足 | `welcome.mp3` 较大或分区已接近满 | `fwrite()` 失败并删除目标文件 | 下载前后查看 SPIFFS 使用量 |
| PSRAM 不可用或不足 | PSRAM 初始化失败、内存碎片严重 | 返回 `ESP_ERR_NO_MEM` | 观察 `failed to allocate PSRAM buffer` 日志 |
| 下载中播放同一文件 | 下载未完成时发送 `0x92` | 播放失败或播放旧文件 | 下载完成日志出现后再播放 |
| 并发下载被拒绝 | 下载中重复发送 `0x91` | 返回 busy，第二次请求不会执行 | 连续发送两次 `0x91` 验证日志 |

## 迁移步骤

1. 烧录新固件。
2. 确保设备已连接 WiFi，且能访问：
   ```text
   http://192.168.26.3:8080/welcome.mp3
   ```
3. 通过串口发送下载指令：
   ```text
   BB 04 91 2E
   ```
4. 等待日志出现：
   ```text
   MP3 download task finished: /spiffs/welcome.mp3
   ```
5. 通过串口发送播放指令：
   ```text
   BB 04 92 2D
   ```
6. 观察是否播放 `welcome.mp3`。

## 后续建议

1. 为 `app_mp3_download_start_async()` 增加串口 ACK，例如 `accepted`、`busy`、`failed`。
2. 将 URL 和目标文件名通过串口载荷下发，而不是写死在 `app_usart_protocol.c`。
3. 下载完成后可以自动播放，或回传下载完成状态让上位机再发 `0x92`。
4. 若后续需要多个离线音频文件，建议增加文件名编号映射和剩余空间检查。
